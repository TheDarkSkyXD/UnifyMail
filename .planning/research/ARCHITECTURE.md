### Component Responsibilities

| Component | Responsibility | Implementation |
|-----------|----------------|----------------|
| `index.js` (generated) | Platform-aware binary loading with local fallback | Generated by `napi build --platform` |
| `index.d.ts` (generated) | TypeScript declarations for all 5 exports | Auto-generated by `#[napi]` macros |
| `lib.rs` | Module entry point, init hook, re-exports | `#[napi(module_exports)]` fn |
| `provider.rs` | Parse providers.json, sync lookup by email domain | `OnceLock<Vec<Provider>>`, serde_json |
| `imap.rs` | Async IMAP connection test, capability detection | `#[napi] async fn`, async-imap, tokio |
| `smtp.rs` | Async SMTP connection test with TLS/STARTTLS | `#[napi] async fn`, lettre, tokio |
| `validator.rs` | Full account validation (MX+IMAP+SMTP) | `#[napi] async fn`, orchestrates imap+smtp |
| Tokio runtime | Thread pool for all async operations | Managed internally by napi-rs `async` feature |

## Recommended Project Structure

```
app/mailcore/
├── Cargo.toml              # Rust manifest: crate-type=cdylib, napi features
├── build.rs                # napi-build setup (required, one-liner)
├── src/
│   ├── lib.rs              # Module root: #[napi(module_exports)] init, re-exports
│   ├── provider.rs         # providerForEmail + registerProviders (sync)
│   ├── imap.rs             # testIMAPConnection (async)
│   ├── smtp.rs             # testSMTPConnection (async)
│   └── validator.rs        # validateAccount (async, composes imap+smtp)
├── resources/
│   └── providers.json      # 500+ provider DB (unchanged from C++ version)
├── package.json            # name=mailcore-napi, scripts, napi config section
├── index.js                # Generated: platform binary loader (do not edit)
└── index.d.ts              # Generated: TypeScript types (do not edit)
```

### Structure Rationale

- **Single `src/` level:** No nested modules needed — 5 functions across 4 files is shallow enough that a flat layout is clearest.
- **`provider.rs` separate from `validator.rs`:** Provider lookup is sync; validation is async. Keeping them separate avoids mixing runtime concerns.
- **`build.rs` at root:** Required by napi-build; must exist alongside Cargo.toml. One call: `napi_build::setup()`.
- **`index.js` / `index.d.ts` in package root:** napi CLI outputs them here. Committed to repo so consumers don't need to build Rust to get type definitions.
- **`resources/providers.json` stays in `mailcore/`:** Already there; Rust code resolves it relative to the `.node` binary location at runtime.

## Architectural Patterns

### Pattern 1: napi-rs #[napi] Macro — Sync Export

**What:** Mark a Rust function with `#[napi]` to export it as a Node.js function. napi-rs handles all marshaling and type generation automatically.

**When to use:** `providerForEmail` and `registerProviders` — fast in-memory operations, no I/O.

**Trade-offs:** Zero overhead. Return `Result<Option<T>>` to express nullable return. napi-rs converts `Err` to JS `Error` automatically.

**Example:**
```rust
// src/provider.rs
use napi_derive::napi;
use napi::Result;

#[napi]
pub fn provider_for_email(email: String) -> Result<Option<MailProviderInfo>> {
    let providers = PROVIDERS.get().ok_or_else(|| {
        napi::Error::from_reason("Providers not initialized — call registerProviders first")
    })?;
    Ok(providers.find_by_email(&email))
}

#[napi(js_name = "registerProviders")]
pub fn register_providers(json_path: String) -> Result<()> {
    let data = std::fs::read_to_string(&json_path)
        .map_err(|e| napi::Error::from_reason(e.to_string()))?;
    let providers = parse_providers(&data)?;
    PROVIDERS.set(providers).ok();
    Ok(())
}
```

The `#[napi]` macro converts `snake_case` to `camelCase` in the JS output automatically. `provider_for_email` becomes `providerForEmail` in the generated `index.d.ts`.

### Pattern 2: napi-rs #[napi] async fn — Async Export with Tokio

**What:** Mark an `async fn` with `#[napi]` and enable the `async` feature. napi-rs manages a tokio runtime internally and converts the returned future to a JS Promise.

**When to use:** `testIMAPConnection`, `testSMTPConnection`, `validateAccount` — all require network I/O.

**Trade-offs:** Node.js event loop is never blocked. The tokio runtime is created once when the first async napi function is called and shut down via a Node.js cleanup hook when the process exits. No manual `Runtime::new()` needed.

**Example:**
```rust
// src/imap.rs
use napi_derive::napi;
use napi::Result;

#[napi(object)]
pub struct ImapConnectionOpts {
    pub hostname: String,
    pub port: u16,
    pub connection_type: Option<String>,  // "tls" | "starttls" | "clear"
    pub username: Option<String>,
    pub password: Option<String>,
    pub oauth2_token: Option<String>,
}

#[napi(object)]
pub struct ImapConnectionResult {
    pub success: bool,
    pub error: Option<String>,
    pub capabilities: Option<Vec<String>>,
}

#[napi]
pub async fn test_imap_connection(opts: ImapConnectionOpts) -> Result<ImapConnectionResult> {
    // Runs in tokio runtime — does NOT block Node.js event loop
    match connect_imap(&opts).await {
        Ok(caps) => Ok(ImapConnectionResult {
            success: true,
            error: None,
            capabilities: Some(caps),
        }),
        Err(e) => Ok(ImapConnectionResult {
            success: false,
            error: Some(e.to_string()),
            capabilities: None,
        }),
    }
}
```

Generated TypeScript: `export function testImapConnection(opts: ImapConnectionOpts): Promise<ImapConnectionResult>`

**Critical:** The existing type definitions use `testIMAPConnection` (all-caps IMAP). Use `#[napi(js_name = "testIMAPConnection")]` to preserve the original naming exactly, because the consumers in `onboarding-helpers.ts` and `mailsync-process.ts` use those exact names.

### Pattern 3: OnceLock for Module-Level State (Provider Database)

**What:** Use `std::sync::OnceLock<T>` (stable Rust 1.70+) for shared state initialized at module load and read-only afterward. This replaces the C++ `MailProvidersManager::sharedManager()` singleton.

**When to use:** The providers database — loaded once from JSON at module initialization, then queried on every `providerForEmail` call (potentially thousands of times per session).

**Trade-offs:** OnceLock is the modern replacement for `lazy_static` and `once_cell`. It is `Send + Sync`, requires no external crates, and provides zero-cost reads after initialization. Thread-safe without locking on the read path.

**Example:**
```rust
// src/provider.rs
use std::sync::OnceLock;

static PROVIDERS: OnceLock<ProviderDatabase> = OnceLock::new();

// Called from the module_init hook:
pub fn init_providers(json_path: &str) -> Result<(), Box<dyn std::error::Error>> {
    let data = std::fs::read_to_string(json_path)?;
    let db = ProviderDatabase::from_json(&data)?;
    PROVIDERS.set(db).map_err(|_| "Providers already initialized".into())
}
```

### Pattern 4: Module Init Hook (#[napi(module_exports)])

**What:** A function annotated `#[napi(module_exports)]` runs synchronously when the `.node` file is first `require()`'d. Used to initialize the provider database from providers.json automatically, replicating the C++ `addon.cpp` `Init()` behavior.

**When to use:** One-time setup that must complete before any function can be called.

**Trade-offs:** Runs on the Node.js main thread. Must be fast and synchronous. Parsing 39KB of JSON is well within acceptable synchronous startup cost (~1ms).

**Example:**
```rust
// src/lib.rs
use napi_derive::napi;
use std::path::PathBuf;

#[napi(module_exports)]
pub fn init() {
    // Resolve providers.json relative to this .node file's directory
    let addon_path = get_current_dll_path();   // platform-specific
    let providers_path = addon_path
        .parent()
        .unwrap_or(&PathBuf::from("."))
        .join("resources")
        .join("providers.json");

    if let Err(e) = provider::init_providers(&providers_path.to_string_lossy()) {
        eprintln!("[mailcore-napi] Warning: could not load providers: {}", e);
    }
}
```

The `get_current_dll_path()` helper uses `dladdr` (Unix) or `GetModuleHandleEx`/`GetModuleFileName` (Windows) — the same approach as the existing `GetProvidersJsonPath()` in C++.

## Data Flow

### Sync Flow: providerForEmail

```
onboarding-helpers.ts
  require('mailcore-napi')       — loads index.js → finds .node for current platform
  providerForEmail('user@gmail.com')
    ↓ napi-rs marshaling (String → Rust String)
  provider_for_email(email: String) in provider.rs
    ↓ PROVIDERS.get() — zero-cost static lookup
  ProviderDatabase::find_by_email(&email)
    ↓ domain suffix match → mxMatch regex (all in-memory)
  Option<MailProviderInfo>
    ↓ napi-rs marshaling (Rust struct → JS Object)
  { identifier, servers: { imap, smtp, pop }, domainMatch, mxMatch }
```

### Async Flow: testIMAPConnection

```
onboarding-helpers.ts (or mailsync-process.ts)
  testIMAPConnection({ hostname, port, connectionType, ... })
    ↓ napi-rs: captures args, returns Promise to JS immediately
  [tokio runtime thread pool]
    async_imap::connect(hostname:port) via TcpStream
    TLS handshake (tokio-native-tls or rustls)
    Parse CAPABILITY response → Vec<String>
    ↓
  Result<ImapConnectionResult>
    ↓ napi-rs: resolves the Promise on main thread
  Promise<{ success, error?, capabilities? }>
```

### Module Load Flow

```
require('mailcore-napi')
  ↓ index.js
  platform detection (process.platform + process.arch)
  existsSync('./mailcore-napi.win32-x64-msvc.node') → true (local build)
  require('./mailcore-napi.win32-x64-msvc.node')
    ↓ Node.js dlopen()
    napi_register_module_v1() called by Node.js
    #[napi(module_exports)] fn init() runs:
      - Resolve providers.json path
      - Parse 39KB JSON → ProviderDatabase
      - PROVIDERS.set(db)
      - tokio runtime NOT started yet (lazy — first async call triggers it)
    Exports object populated with all 5 functions
  module.exports = { providerForEmail, registerProviders,
                     validateAccount, testIMAPConnection, testSMTPConnection }
```

### .node File Naming Convention

`napi build --platform` produces files named:
- `mailcore-napi.win32-x64-msvc.node` (Windows x64)
- `mailcore-napi.darwin-x64.node` (macOS Intel)
- `mailcore-napi.darwin-arm64.node` (macOS Apple Silicon)
- `mailcore-napi.linux-x64-gnu.node` (Linux x64 glibc)
- `mailcore-napi.linux-arm64-gnu.node` (Linux ARM64 glibc)

The generated `index.js` uses `process.platform` + `process.arch` to select and `require()` the correct file. It also tries a local `.node` file first (for development builds), then falls back to optionally-installed npm packages (for distribution).

### Cross-Platform Build Order

For CI/CD producing prebuilt binaries per platform:

```
1. Windows runner:   napi build --platform --release --target x86_64-pc-windows-msvc
2. macOS runner:     napi build --platform --release --target x86_64-apple-darwin
                     napi build --platform --release --target aarch64-apple-darwin

                     napi build --platform --release --target aarch64-unknown-linux-gnu

```

The resulting `.node` file is placed in `app/mailcore/` alongside `index.js`. The `require('mailcore-napi')` call in `app/package.json` with `"file:mailcore"` resolves to this directory and finds `index.js`, which loads the `.node` file.

## Electron ABI Compatibility

**Finding:** N-API (the interface used by napi-rs) is ABI-stable across Node.js versions and across Electron versions that ship the same Node.js. Electron 39 includes Node.js 22.21.1. A napi-rs addon compiled against napi4 (N-API version 4) is guaranteed to work on any platform with N-API version 4 or higher — which includes all Electron versions from Electron 3+ (Node.js 10+).

**No electron-rebuild required:** Unlike node-addon-api (C++) addons, napi-rs addons do not need to be rebuilt per Electron version. The N-API guarantee applies. This is one of the core motivations for the rewrite.

**Known historical issue (resolved):** napi-rs 2.2.1 introduced a Windows regression with `thread_local!` that broke Electron (issue #1175). This was fixed in 2.2.2+ via PR #1176. Using napi-rs v3 (the current version) avoids this entirely.

## Anti-Patterns

### Anti-Pattern 1: Calling Node.js APIs from the Tokio Thread Pool

**What people do:** Try to use `napi::Env` inside an `async fn` body to call back into JavaScript mid-operation.

**Why it's wrong:** `napi::Env` is not `Send`. N-API calls from non-main threads crash Node.js with "must be called from the main thread" assertion failures. Only the main Node.js thread can touch the env.

**Do this instead:** Collect all results in plain Rust types (`String`, `Vec<String>`, `bool`) within the `async fn`, return them from the function, and let napi-rs marshal the result back to JS on the main thread. For progress callbacks, use `napi::threadsafe_function::ThreadsafeFunction`.

### Anti-Pattern 2: Blocking Inside async fn

**What people do:** Call synchronous blocking I/O (e.g., `std::net::TcpStream::connect`) inside an `async fn`.

**Why it's wrong:** Blocks the tokio worker thread that was executing the future. Under load, all tokio threads can become blocked, preventing new async operations from starting.

**Do this instead:** Use async equivalents: `tokio::net::TcpStream::connect`, `async-imap`, `lettre`'s async transport. Wrap unavoidable blocking calls with `tokio::task::spawn_blocking`.

### Anti-Pattern 3: Using lazy_static or once_cell for Shared State

**What people do:** Pull in `lazy_static` or `once_cell` crates for module-level statics.

**Why it's wrong:** Not wrong per se, but unnecessary since Rust 1.70. Adds crate dependencies for something the standard library now provides.

**Do this instead:** Use `std::sync::OnceLock<T>` (stable since Rust 1.70, included in std). For values that initialize at first use rather than being set explicitly, use `std::sync::LazyLock<T>` (stable since Rust 1.80).

### Anti-Pattern 4: Hardcoding providers.json Path

**What people do:** Use a relative path like `"./resources/providers.json"` resolved from the process working directory.

**Why it's wrong:** Electron's working directory varies by how the app is launched, packaged, and run in dev vs production. The C++ addon solved this by using `GetModuleFileNameA` / `dladdr` to find the path of the `.node` file itself, then navigating from there.

**Do this instead:** Resolve the providers.json path relative to the `.node` file location using `dladdr` (Unix) or `GetModuleHandleEx` + `GetModuleFileName` (Windows). Alternatively, embed the providers.json contents directly in the binary using `include_str!("../resources/providers.json")`, which eliminates the runtime path resolution entirely and is the cleanest approach for a 39KB file.

```rust
// Simplest approach: embed at compile time
static PROVIDERS_JSON: &str = include_str!("../resources/providers.json");

static PROVIDERS: OnceLock<ProviderDatabase> = OnceLock::new();

pub fn ensure_providers_initialized() {
    PROVIDERS.get_or_init(|| {
        ProviderDatabase::from_json(PROVIDERS_JSON)
            .expect("Failed to parse embedded providers.json")
    });
}
```

This eliminates the `registerProviders(jsonPath)` runtime path resolution problem entirely. The `registerProviders` export can still be implemented for API compatibility (it re-parses from the given path), but the automatic initialization on module load uses the embedded data.

## Scaling Considerations

This is a desktop addon. Scaling is not a concern in the traditional sense. The relevant "scale" dimension is concurrent async operations per user session.

| Scale | Architecture Impact |
|-------|---------------------|
| 1-3 accounts (typical) | Default tokio thread pool (num_cpus threads) is excessive; fine |
| 10+ accounts onboarding | Multiple concurrent `validateAccount` calls — tokio handles naturally |
| Provider lookup calls | Millions per session are fine — OnceLock + Vec iteration is O(n), n=500 |
