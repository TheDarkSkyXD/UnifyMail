<objective>
Implement the core sync algorithms: CONDSTORE incremental sync with modseq change detection and truncation, UID-range fallback for non-CONDSTORE servers, UIDVALIDITY change handling (RFC 4549 full re-sync), folder priority ordering, and per-operation timeout wrapping.

Purpose: These algorithms determine which messages are new/changed on each sync cycle. They form the inner loop of the background sync worker. CONDSTORE is the primary strategy (~90% of modern servers); UID-range is the fallback.
Output: Tested sync algorithms that work with the ImapSession from Plan 03 and mail_processor from Plan 04.

Implement the core sync algorithm functions in `app/mailsync-rs/src/imap/sync_worker.rs`:

1. **`async fn sync_folder(session: &mut ImapSession, folder: &mut Folder, store: &MailStore, delta: &DeltaStream, account: &Account) -> Result<bool, SyncError>`** — Top-level folder sync dispatcher:
   - Call `session.select_condstore(&folder.path)` with 30s timeout
   - Get server uidvalidity from mailbox
   - Get stored state via `get_sync_state(folder)`
   - Check UIDVALIDITY change: if `stored.uidvalidity != 0 && stored.uidvalidity != server_uidvalidity`, call `handle_uidvalidity_change()`
   - Check CONDSTORE: if `mailbox.highest_modseq.is_some()`, call `sync_folder_condstore()`
   - Otherwise: call `sync_folder_uid_range()`
   - Update stored sync state after sync
   - Return true if any messages were new/changed

2. **`async fn sync_folder_condstore(...) -> Result<bool, SyncError>`** — CONDSTORE incremental sync:
   - Compare server modseq + uidnext against stored values
   - If both match -> return Ok(false) (no changes)
   - If delta > MODSEQ_TRUNCATION_THRESHOLD: calculate truncated UID range `"(uidnext-12000):*"`
   - Otherwise: use `"1:*"`
   - Build query: `"(UID FLAGS ENVELOPE BODYSTRUCTURE) (CHANGEDSINCE {stored_modseq})"` (for non-Gmail) or `"(UID FLAGS ENVELOPE BODYSTRUCTURE X-GM-LABELS X-GM-MSGID X-GM-THRID) (CHANGEDSINCE {stored_modseq})"` (for Gmail)
   - Call `session.uid_fetch(uid_range, query)` with 120s timeout
   - Iterate stream with 30s per-item timeout
   - For each fetch: call `process_fetched_message()` from mail_processor
   - Save messages and threads to store, emit deltas
   - Update folder sync state: new modseq, uidnext

3. **`async fn sync_folder_uid_range(...) -> Result<bool, SyncError>`** — UID range fallback:
   - Use `"1:*"` or `"{synced_min_uid}:*"` based on stored state
   - Fetch UIDs and flags to detect new/changed messages
   - Process new messages via `process_fetched_message()`
   - Detect deleted messages: UIDs in local DB but not in server response
   - Update folder sync state

4. **`async fn handle_uidvalidity_change(...) -> Result<(), SyncError>`** — RFC 4549:
   - Log warning about UIDVALIDITY change
   - Unlink all messages in folder (set remote_uid to 0 or delete from folder association)
   - Trigger full UID range re-sync
   - Increment uidvalidity_reset_count
   - Update all sync state fields to new server values

5. **`async fn run_sync_cycle(session: &mut ImapSession, store: &MailStore, delta: &DeltaStream, account: &Account) -> Result<bool, SyncError>`** — One complete sync cycle:
   - Enumerate folders via `session.list_folders(account)` (or use cached list)
   - Sort by role priority
   - For each folder: call `sync_folder()`
   - Return true if any folder had changes

6. **Per-operation timeout pattern:** Every IMAP operation call follows:
```rust
tokio::time::timeout(Duration::from_secs(N), session.method(...))
    .await
    .map_err(|_| SyncError::Timeout)?
    .map_err(SyncError::from)?
```
