Implement in `app/mailsync-rs/src/imap/mail_processor.rs`:

1. **`decode_mime_header(raw: &[u8]) -> String`** -- Decodes RFC 2047 MIME encoded-words using `rfc2047_decoder::decode()`. Falls back to `String::from_utf8_lossy()` on error. CRITICAL: Used for subject and message-id fields. NOT for address mailbox/host parts (those are raw bytes in C++).

2. **`id_for_message(account_id: &str, folder_path: &str, uid: u32, envelope: &Envelope) -> String`** -- Exact C++ MailUtils::idForMessage() Scheme v1:
   - Decode subject via `decode_mime_header()`
   - Decode message-id via `decode_mime_header()`
   - Extract recipients from To+CC+BCC: `mailbox@host` format (raw bytes, not decoded)
   - Sort recipients lexicographically, join without separator
   - Parse date: `chrono::DateTime::parse_from_rfc2822(date_str)` -> Unix timestamp string. If None or parse fails -> `"folder_path:uid"` fallback
   - Build input: `"{account_id}-{timestamp}{subject}-{recipients}-{message_id}"`
   - SHA-256 hash, take first 30 bytes, encode with `bs58::encode().with_alphabet(bs58::Alphabet::BITCOIN)`

3. **`gmail_thread_id(fetch: &async_imap::types::Fetch) -> Option<u64>`** -- Extract X-GM-THRID from fetch response. Try to access via `fetch.parsed()` or internal response field. If `fetch.response` is private, use an alternative approach: parse the raw response bytes looking for the GmailThrId attribute value (see Pattern 5 in research). As a last resort, include X-GM-THRID in the fetch query and extract from the response attributes.

4. **`extract_gmail_extensions(fetch: &async_imap::types::Fetch) -> GmailExtensions`** -- Extract all three Gmail extension attributes:
   ```rust
   pub struct GmailExtensions {
       pub labels: Vec<String>,    // from fetch.gmail_labels()
       pub msg_id: Option<u64>,    // from fetch.gmail_msg_id()
       pub thr_id: Option<u64>,    // from gmail_thread_id()
   }
   ```

5. **`pub const GMAIL_SKIP_SENT_APPEND: bool = true`** -- Constant documenting GMAL-04 (Gmail auto-saves sent mail, APPEND causes duplicates). Phase 8 SendDraftTask checks this flag. Include doc comment explaining the behavior.

6. **Labels storage note:** X-GM-LABELS are stored as `Vec<String>` on the Message.labels field, which is serialized as a JSON array in the Message record's `labels` column. This matches the existing Message model from Phase 6, which already has a `labels: Vec<String>` field. The CONTEXT.md mentioned a MessageLabel join table approach, but the existing Phase 6 Message model already provides a labels field -- using it avoids schema changes beyond Phase 7's scope.

For stable ID tests: Compute expected IDs manually using the algorithm (SHA-256 of known input, Base58 encode first 30 bytes) and hardcode as expected values in tests. This ensures byte-for-byte compatibility.

For Gmail extension tests: These require constructing mock Fetch responses. If async-imap Fetch cannot be easily constructed in tests, use the raw attribute parsing approach and test the extraction logic on known byte patterns.

Continue in `app/mailsync-rs/src/imap/mail_processor.rs`:

1. **`process_fetched_message(fetch: &Fetch, folder: &Folder, account: &Account, is_gmail: bool) -> Result<(Message, Option<Thread>), SyncError>`** -- Converts an async-imap Fetch response into a Message model:
   - Extract UID from fetch.uid
   - Extract envelope from fetch.envelope()
   - Generate stable ID via `id_for_message()`
   - Populate Message fields from envelope: subject (decoded), date, header_message_id, from/to/cc/bcc (as JSON contact objects `{"name": "", "email": ""}`)
   - Set remote_uid, remote_folder_id = folder.id
   - Extract flags: \Seen -> !unread, \Flagged -> starred, \Draft -> draft
   - If is_gmail: extract Gmail extensions, set g_msg_id, g_thr_id, labels (Vec<String> from X-GM-LABELS stored directly on Message.labels field)
   - **Threading:**
     - Gmail: thread_id = format!("{:x}", gmail_thr_id) (hex string of X-GM-THRID)
     - Non-Gmail: thread_id derived from References/In-Reply-To headers. Algorithm: take the last message-id in References header (or In-Reply-To if no References), hash it with account_id using SHA-256+Base58 to get thread_id. If no threading headers, use the message's own stable ID as thread_id (single-message thread).
   - Create Thread record if this thread_id is new (caller checks DB)
   - Return (Message, Option<Thread>) where Thread is Some if new thread created

2. **`BodyQueue` struct** -- Priority queue for body fetch requests:
   ```rust
   pub struct BodyQueue {
       queue: VecDeque<String>,
   }
   impl BodyQueue {
       pub fn new() -> Self;
       pub fn enqueue_priority(&mut self, ids: Vec<String>);  // FIFO with dedup, front insert
       pub fn enqueue_background(&mut self, id: String);      // Append at back with dedup
       pub fn next(&mut self) -> Option<String>;              // Pop front
       pub fn is_empty(&self) -> bool;
       pub fn len(&self) -> usize;
   }
   ```
   - `enqueue_priority()`: Insert each ID at the front (reversed so order is preserved). Skip if already in queue.
   - `enqueue_background()`: Append at back. Skip if already in queue.
   - `next()`: Pop from front.

3. **`parse_flags(flags: &[Flag]) -> (bool, bool, bool)`** -- Parse IMAP flags to (unread, starred, draft):
   - \Seen -> unread=false (inverse!)
   - \Flagged -> starred=true
   - \Draft -> draft=true
   - Default: unread=true, starred=false, draft=false

Write tests for message conversion using manually constructed test data (not live IMAP). For threading, test both Gmail (hex thread_id) and non-Gmail (References-based) paths. For BodyQueue, test priority insertion, dedup, and FIFO ordering.
