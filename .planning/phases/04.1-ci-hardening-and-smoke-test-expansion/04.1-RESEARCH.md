## Summary

Phase 4.1 is a pure gap-closure phase: no new Rust code, no new API surface, no dependency changes. The work is entirely in CI workflow YAML files, a new Node.js smoke script, and a stale file deletion.

The existing smoke tests in all 4 workflows (`build-linux.yaml`, `build-linux-arm64.yaml`, `build-macos.yaml`, `build-windows.yaml`) are inline Node.js one-liners or multi-line blocks that only test `providerForEmail`. They must be replaced with a call to a shared `app/mailcore-rs/test/smoke.js` script that checks all 5 exports.

The macOS workflow (`build-macos.yaml`) has a verified bug on line 34: its `actions/cache@v4` key references `hashFiles('yarn.lock')` but the project uses npm (the root `package-lock.json` exists; no `yarn.lock` exists). This means the macOS Node modules cache never hits, adding unnecessary install time on every macOS CI run.

The `app/mailcore-rs/index.js` file is the auto-generated napi-rs loader from the initial scaffold. It was superseded by the hand-written `loader.js` in Phase 1. The `package.json` `main` field already correctly points to `loader.js`. The stale `index.js` is dead code that creates confusion about the canonical entry point and should be deleted.

**Primary recommendation:** Create `app/mailcore-rs/test/smoke.js`, replace all 4 CI smoke test inline commands with `node app/mailcore-rs/test/smoke.js`, fix the macOS cache key, and delete `index.js`.

### Pattern 1: Shared Cross-Platform Node.js Smoke Script

**What:** A single `smoke.js` called identically from all 4 CI workflows. Uses `require('mailcore-napi')` (resolves via npm workspace symlink to `app/mailcore-rs/loader.js`). Performs type-checks and one functional check with verbose per-check logging.

**When to use:** When the same test command must work on bash (Linux, macOS) and PowerShell (Windows) without shell-specific quoting tricks.

**Why shared script over inline:** The Windows CI inline smoke test is already a compressed one-liner to avoid PowerShell quoting issues with multi-line `-e` strings. A shared file removes this constraint and makes the test maintainable in one place.

**Example (smoke.js):**
```javascript
// app/mailcore-rs/test/smoke.js
// Cross-platform smoke test: verifies all 5 exports of mailcore-napi.
// Called from CI: node app/mailcore-rs/test/smoke.js
// Working directory: app/ (so require('mailcore-napi') resolves correctly)
'use strict';

// require path must work from app/ working directory
const m = require('mailcore-napi');

let failed = false;

// providerForEmail: functional check (synchronous, embedded data)
const result = m.providerForEmail('test@gmail.com');
if (!result || typeof result !== 'object') {
  console.error('FAIL: providerForEmail did not return an object');
  failed = true;
} else {
  console.log('PASS: providerForEmail returned result:', result.displayName || JSON.stringify(result));
}

// registerProviders: type-check only (auto-loaded at import time)
if (typeof m.registerProviders !== 'function') {
  console.error('FAIL: registerProviders is not a function');
  failed = true;
} else {
  console.log('PASS: registerProviders is function');
}

// testIMAPConnection: type-check only (async network, no server in CI)
if (typeof m.testIMAPConnection !== 'function') {
  console.error('FAIL: testIMAPConnection is not a function');
  failed = true;
} else {
  console.log('PASS: testIMAPConnection is function');
}

// testSMTPConnection: type-check only (async network, no server in CI)
if (typeof m.testSMTPConnection !== 'function') {
  console.error('FAIL: testSMTPConnection is not a function');
  failed = true;
} else {
  console.log('PASS: testSMTPConnection is function');
}

// validateAccount: type-check only (async network, no server in CI)
if (typeof m.validateAccount !== 'function') {
  console.error('FAIL: validateAccount is not a function');
  failed = true;
} else {
  console.log('PASS: validateAccount is function');
}

if (failed) {
  process.exit(1);
}
console.log('All smoke tests passed.');
```

### Pattern 3: macOS Cache Key Fix

`build-macos.yaml` line 34. Change:
```yaml
key: ${{ matrix.os }}-${{ runner.arch }}-deps-${{ hashFiles('yarn.lock') }}-${{ hashFiles('app/package-lock.json') }}
```
To:
```yaml
key: ${{ matrix.os }}-${{ runner.arch }}-deps-${{ hashFiles('package-lock.json') }}-${{ hashFiles('app/package-lock.json') }}
```

The `hashFiles()` function in GitHub Actions takes a path relative to the workspace root. `package-lock.json` exists at the workspace root. `yarn.lock` does not exist, so `hashFiles('yarn.lock')` returns the same empty-string hash on every run, making the first segment of the key meaningless (the cache only varies on `app/package-lock.json` changes, not root changes).

### Anti-Patterns to Avoid

- **Shell-specific inline test code:** Avoid `-e` multi-line strings with embedded quotes in CI. They require escaping that differs between PowerShell (Windows) and bash (Linux/macOS). The shared `.js` file eliminates this entirely.
- **Mixing binary-check and cross-platform checks:** The 8MB binary size step uses `du`, `ls`, and `cut` â€” POSIX shell tools unavailable on Windows. It must stay as its own Linux-only step. Do not put binary size checking in `smoke.js`.
- **Running smoke test before addon build:** The smoke test step must remain after the napi-rs build step in each workflow. The current position (after build, before lint) is correct.
- **Removing `working-directory: app`:** The require resolution path `require('mailcore-napi')` depends on `app/node_modules/mailcore-napi` existing as an npm symlink. Changing working directory to the repo root would break this.

## Common Pitfalls

### Pitfall 1: smoke.js require path assumes wrong working directory

**What goes wrong:** If `working-directory` is set to the repo root instead of `app/`, the `require('mailcore-napi')` call fails with "Cannot find module" because the npm workspace symlink lives at `app/node_modules/mailcore-napi`.

**Why it happens:** The CI step `working-directory: app` was intentional in Phase 4. If someone "simplifies" the step by removing `working-directory`, the require resolution breaks.

**How to avoid:** Keep `working-directory: app` on the smoke test step in all 4 workflows. Alternatively, write `smoke.js` using `path.resolve(__dirname, '../../node_modules/mailcore-napi')` â€” but this is more fragile. The `working-directory` approach is established project convention.

**Warning signs:** CI error `Error: Cannot find module 'mailcore-napi'` on the smoke step.

### Pitfall 2: macOS matrix â€” cache key applies to both arch entries

**What goes wrong:** The `build-macos.yaml` matrix has two entries (`arm64` and `x64`). The `Cache NodeJS modules` step uses `${{ matrix.os }}-${{ runner.arch }}` in the key, so each arch gets its own cache entry. The `yarn.lock` bug affects both entries identically. Both must be fixed by updating the single cache step (the matrix handles the per-arch differentiation automatically).

**How to avoid:** Edit the one cache key in the `build-macos.yaml` jobs.build-macos.steps cache step. The matrix expansion takes care of the rest.

### Pitfall 3: Deleting index.js could break external callers

**What goes wrong:** If any code outside `app/mailcore-rs/` does `require('./index.js')` or `require('../mailcore-rs/index.js')` explicitly (bypassing the `main` field), deleting `index.js` would break it.

**Why it happens:** Stale imports sometimes reference specific file paths instead of the package name.

**How to avoid:** Before deleting, search the repo for explicit `index.js` references under `app/mailcore-rs/`. Project history confirms `loader.js` has been the canonical entry since Phase 1 and the `package.json` main is already `loader.js`. All consumer code (`onboarding-helpers.ts`, `mailsync-process.ts`) uses `require('mailcore-napi')` â€” the package name, not a path. No explicit path to `index.js` should exist.

**Warning signs:** Any `require` string containing `mailcore-rs/index` in the codebase.

### Pitfall 4: smoke.js placed under `tests/` (Rust test directory)

**What goes wrong:** `app/mailcore-rs/tests/` is Cargo's integration test directory. Placing `smoke.js` there is semantically confusing and risks future Cargo commands treating `.js` files unexpectedly.

**How to avoid:** Use `app/mailcore-rs/test/` (singular) for Node.js tests â€” a new directory that doesn't conflict with Cargo conventions.
