Create `app/mailcore-rs/tests/imap_tests.rs` with a mock IMAP server and comprehensive test coverage.

**Mock IMAP server pattern:**

Use `tokio::net::TcpListener::bind("127.0.0.1:0")` to get a random available port. Each test creates its own listener -- no shared state, no TEST_MUTEX needed.

The mock server is a minimal IMAP responder that:
1. Sends greeting: `* OK IMAP4rev1 Server ready\r\n`
2. Handles LOGIN: reads the LOGIN command, responds `<tag> OK LOGIN completed\r\n` (or `<tag> NO` for auth failure tests)
3. Handles CAPABILITY: responds with configurable capability string `* CAPABILITY IMAP4rev1 IDLE CONDSTORE ...\r\n<tag> OK\r\n`
4. Handles LOGOUT: responds `* BYE\r\n<tag> OK\r\n`

For XOAUTH2 tests, the mock server:
1. Handles AUTHENTICATE XOAUTH2: reads the base64-encoded SASL token on the next line
2. Decodes base64 and validates the format: `user=<email>\x01auth=Bearer <token>\x01\x01`
3. Responds OK if format is valid, NO if invalid

**IMPORTANT:** Tests call the internal `do_test_imap()` function directly (it is `pub`), NOT the `#[napi]`-decorated function (which requires a Node.js runtime). Wrap calls in `tokio::time::timeout()` as the napi function would.

**Test for TLS connections:**
Since the mock server is a plain TCP server, TLS tests need a self-signed certificate. For Phase 2 (strict cert validation only, no allowInsecureSsl), TLS connection tests against a mock server with self-signed certs will fail with `tls_error`. This is expected behavior. Test that:
- The TLS error is correctly classified as `error_type: "tls_error"`
- The error message includes the hostname and port

For a positive TLS test, test the clear connection path which exercises the same auth+capability logic without TLS complexity. The TLS handshake itself is tested by the `tls_error` classification test and will be validated against real servers in Phase 3 integration.

**Alternative approach for TLS positive test:** If the executor can generate a self-signed cert and configure rustls to trust it in a test-only TLS config, do so. Otherwise, rely on clear-connection tests for auth+capability logic and TLS-error tests for TLS classification. Use Claude's discretion per CONTEXT.md.

```rust
#[tokio::test]
async fn test_clear_connection_with_password() {
    let (port, _handle) = start_mock_imap_server("IDLE CONDSTORE", MockAuthMode::AcceptPassword).await;
    let opts = IMAPConnectionOptions {
        hostname: "127.0.0.1".to_string(),
        port: port as u32,
        connection_type: Some("clear".to_string()),
        username: Some("user@example.com".to_string()),
        password: Some("password123".to_string()),
        oauth2_token: None,
    };
    let result = do_test_imap(&opts).await.unwrap();
    assert!(result.success);
    assert!(result.capabilities.unwrap().contains(&"idle".to_string()));
}
```
