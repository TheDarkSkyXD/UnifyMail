Extend MailStore with generic CRUD operations (save, remove, find, find_all, count), add a separate reader connection for WAL concurrency, implement MailStoreTransaction with delta accumulation, and wire store operations to the existing delta emission pipeline.

Purpose: This is the data access backbone. Every subsequent phase (IMAP sync, task execution, CalDAV/CardDAV) will call these methods to persist and query models. The single-writer + concurrent-reader pattern prevents SQLITE_BUSY errors during sync while maintaining data integrity.

**New constructor** `open_with_delta()`:
```rust
pub async fn open_with_delta(
    config_dir: &str,
    delta_stream: Arc<DeltaStream>,
) -> Result<Self, SyncError>
```
Opens two connections (writer + reader) to the same `edgehill.db`. Both connections get the same PRAGMA setup (WAL, page_size, cache_size, synchronous, busy_timeout). Migrations run on the writer only. The `delta_stream` is stored for emit calls. busy_timeout must be 5000ms per CONTEXT.md requirement (update from Phase 5's 10s to match C++ behavior and DATA-01 spec).

Keep existing `open()` for backward compatibility (offline modes don't need reader or delta). Refactor to share PRAGMA setup logic.

**Generic save method:**
```rust
pub async fn save<T: MailModel>(&self, model: &mut T) -> Result<(), SyncError>
```
1. Call `model.increment_version()`
2. Serialize: `let data_json = serde_json::to_string(&model.to_json())?;`
3. Clone all values needed for the move closure (id, account_id, version, data_json, and all indexed column values extracted via a helper)
4. Determine INSERT vs UPDATE: if version == 1, INSERT; else UPDATE (per C++ pattern: version is incremented before this check, so newly created models go from 0->1 = INSERT)
5. Build SQL: `INSERT INTO {table} ({columns}) VALUES (?1,?2,...?N)` or `UPDATE {table} SET {col=?N,...} WHERE id=?1`
6. Execute inside `self.writer.call(move |db| { ... })` using `prepare_cached`
7. After successful write, emit delta: `DeltaStreamItem::new("persist", T::table_name(), vec![model.to_json_dispatch()])`
8. If `self.delta_tx` is Some, call `delta_stream.emit(delta_item)`

**Generic remove method:**
```rust
pub async fn remove<T: MailModel>(&self, model: &T) -> Result<(), SyncError>
```
1. Execute `DELETE FROM {table} WHERE id = ?1` on writer connection
2. Emit delta: `DeltaStreamItem::new("unpersist", T::table_name(), vec![json!({"id": model.id()})])`

**Generic find method:**
```rust
pub async fn find<T: MailModel>(&self, where_clause: &str, params: Vec<Box<dyn rusqlite::types::ToSql + Send>>) -> Result<Option<T>, SyncError>
```
Wait — tokio-rusqlite closures need owned values. Use a different approach for params. The cleanest pattern for tokio-rusqlite is to accept a `Vec<serde_json::Value>` or specific typed params that are Send + 'static.

Better approach: Accept `where_sql: &str` and `params: Vec<String>` (most queries use string params). For numeric params, accept `where_sql` with literals embedded for simple cases, or use a param enum:

```rust

pub enum SqlParam {
    Text(String),
    Int(i64),
    Real(f64),
    Null,
}
```

let sql = format!("SELECT data FROM {} WHERE {} LIMIT 1", T::table_name(), where_sql);

// Deserialize from data column: serde_json::from_str::<T>(&data_str)?

Create `app/mailsync-rs/src/store/transaction.rs`:

The transaction pattern mirrors C++ MailStoreTransaction: a wrapper that gates delta emission.

```rust
pub struct MailStoreTransaction {
    /// Accumulated deltas — emitted on commit, discarded on rollback
    deltas: Vec<DeltaStreamItem>,
    /// Whether commit() has been called
    committed: bool,
    /// Reference back to the store's delta emitter
    delta_stream: Option<Arc<DeltaStream>>,
}
```

**MailStore integration:**

Add to MailStore:
```rust
/// Flag indicating whether a transaction is active.
/// When true, save/remove accumulate deltas in the transaction instead of emitting.
transaction_deltas: Arc<tokio::sync::Mutex<Option<Vec<DeltaStreamItem>>>>,
```

When `transaction_deltas` is `Some(vec)`, save/remove push to the vec instead of emitting directly.

```rust
pub async fn begin_transaction(&self) -> Result<MailStoreTransaction, SyncError>
```
1. Execute `BEGIN IMMEDIATE` on writer connection (IMMEDIATE acquires write lock immediately, matching C++ behavior)
2. Set `transaction_deltas` to `Some(Vec::new())`
3. Return `MailStoreTransaction` handle

**save_in_transaction()** — same as `save()` but pushes delta to the transaction vec instead of emitting.

Actually, better design: make `save()` check internally whether a transaction is active:
- If `transaction_deltas.lock().is_some()`: push delta to the vec
- If `None`: emit directly to delta channel

This avoids needing separate methods for transaction vs non-transaction saves.

**MailStoreTransaction methods:**

```rust
pub async fn commit(mut self, store: &MailStore) -> Result<(), SyncError>
```
1. Execute `COMMIT` on writer connection
2. Take the accumulated deltas from `store.transaction_deltas` (swap with None)
3. Emit each delta to the delta channel
4. Set `self.committed = true`

```rust
pub async fn rollback(mut self, store: &MailStore) -> Result<(), SyncError>
```
1. Execute `ROLLBACK` on writer connection
2. Clear `store.transaction_deltas` (swap with None, drop the vec)
3. Set `self.committed = true` (prevent Drop from trying again)

**Drop impl:**
If `!self.committed`, log a warning and attempt rollback. This is the RAII safety net. Note: Drop cannot be async, so use `std::mem::take` on the delta vec to at least clear it. The SQL ROLLBACK should have happened via explicit call.
