<objective>
Implement afterSave/afterRemove lifecycle hooks for all models that need side effects (metadata join table, ThreadCategory, ThreadCounts, FTS5 indexes), wire the hooks into MailStore.save()/remove(), and add end-to-end pipeline tests verifying the complete save-to-delta flow.

Purpose: The lifecycle hooks maintain secondary tables (join tables, FTS5 search indexes, category counters) that the Electron UI queries. Without these, thread lists show wrong unread counts, search returns no results, and plugin metadata (snooze, read receipts) stops working. This plan completes the MailStore to production readiness.

Output: All lifecycle hooks implemented, FTS5 indexes populated on save, join tables maintained, and end-to-end tests verifying the entire save -> delta -> wire format pipeline matches what Electron expects.
</objective>

Wire `after_save()` and `after_remove()` into `MailStore.save()` and `MailStore.remove()`. These hooks run INSIDE the tokio-rusqlite `writer.call()` closure, using the same `&rusqlite::Connection` that performed the primary INSERT/UPDATE/DELETE. This ensures atomicity â€” if a hook fails, the entire save/remove fails.

**Update MailStore.save():**
After the INSERT/UPDATE statement succeeds (still inside the `call` closure):
1. If `T::supports_metadata()`, call metadata maintenance: `DELETE FROM ModelPluginMetadata WHERE id = ?` then for each metadata entry in model.to_json()["metadata"], `INSERT INTO ModelPluginMetadata (id, accountId, objectType, value, expiration) VALUES (?, ?, ?, ?, ?)`
2. Call `model.after_save(db)` â€” the model-specific hook

**Update MailStore.remove():**
After the DELETE statement (inside `call` closure):
1. If `T::supports_metadata()`, `DELETE FROM ModelPluginMetadata WHERE id = ?`
2. Call `model.after_remove(db)` â€” the model-specific hook

async fn test_save_emits_persist_delta() {
