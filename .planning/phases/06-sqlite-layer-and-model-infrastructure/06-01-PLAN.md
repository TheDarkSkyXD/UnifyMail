Create `app/mailsync-rs/src/models/` directory with the following files:

**mail_model.rs** — Define the `MailModel` trait:
```rust
pub trait MailModel: Serialize + for<'de> Deserialize<'de> + Clone + std::fmt::Debug + Send + 'static {
    fn table_name() -> &'static str;
    fn id(&self) -> &str;
    fn account_id(&self) -> &str;
    fn version(&self) -> i64;
    fn increment_version(&mut self);
    fn to_json(&self) -> serde_json::Value;
    fn to_json_dispatch(&self) -> serde_json::Value { self.to_json() }
    fn columns_for_query() -> &'static [&'static str];
    fn bind_to_statement(&self, stmt: &mut rusqlite::Statement<'_>, data_json: &str) -> rusqlite::Result<()>;
    fn supports_metadata() -> bool { false }
}
```

The `to_json()` default behavior: serialize with serde_json::to_value, then inject `"__cls": Self::table_name()` into the resulting JSON object. Each model may override if needed (e.g., Task where __cls is the task type).

The `bind_to_statement()` takes `data_json: &str` as a pre-serialized JSON string (from `to_json()`). All model types bind at minimum: `id`, `data` (the JSON string), `accountId`, `version`. Then model-specific indexed columns. EXCEPTIONS: Calendar and Event do NOT call the base binding pattern — they bind their own columns directly (no version column).

**Per model struct** — Follow the fat-row pattern from 06-RESEARCH.md Deep Dive: Model Field Mapping section. Each field has:
- Concrete Rust type (String, i64, bool, Option<T>, Vec<serde_json::Value>)
- `#[serde(rename = "...")]` matching exact C++ JSON key from research
- `#[serde(skip_serializing_if = "Option::is_none")]` for optional fields
- `#[serde(default)]` for fields that may be absent in incoming JSON

Critical serde rename rules (MUST match exactly — 06-RESEARCH.md Section "Critical serde rename rules"):
- `account_id` -> `"aid"`
- `version` -> `"v"`
- `header_message_id` -> `"hMsgId"`
- `reply_to_header_message_id` -> `"rthMsgId"`
- `forwarded_header_message_id` -> `"fwdMsgId"`
- `g_msg_id` -> `"gMsgId"`
- `g_thr_id` -> `"gThrId"`
- `synced_at` -> `"_sa"`
- `sync_unsaved_changes` -> `"_suc"`
- `last_message_timestamp` -> `"lmt"` (Thread)
- `first_message_timestamp` -> `"fmt"` (Thread)
- `last_message_sent_timestamp` -> `"lmst"` (Thread)
- `last_message_received_timestamp` -> `"lmrt"` (Thread)
- `calendar_id` -> `"cid"` (Event)
- `recurrence_start` -> `"rs"` (Event)
- `recurrence_end` -> `"re"` (Event)
- `recurrence_id` -> `"rid"` (Event)
- `source` -> `"s"` (Contact)
- `hidden` -> `"h"` (Contact)
- `contact_groups` -> `"gis"` (Contact)
- `book_id` -> `"bid"` (Contact, ContactGroup)
- `google_resource_name` -> `"grn"` (Contact, ContactGroup)

For `bind_to_statement()`, use positional params (?1, ?2, ...) matching `columns_for_query()` order. Use `prepare_cached` pattern (the caller will prepare, this method just binds). The method receives a pre-prepared Statement reference.

**identity.rs** — Plain struct (NOT implementing MailModel). Fields: id, email_address, first_name, last_name, token, created_at. This is the process-level identity from the stdin handshake (already partially in `account.rs` from Phase 5).

**model_plugin_metadata.rs** — Struct with fields: id, account_id, object_type, value (pluginId), expiration. This is NOT a fat-row model (no `data` column). It implements a simpler set of DB operations (not the full MailModel trait).

**task_model.rs** — Named `task_model.rs` (not `task.rs`) to avoid conflict with Rust's `task` keyword associations. The Task model's `__cls` is the task type name (e.g., "SendDraftTask"), pre-set in the data. The `to_json()` implementation should NOT override `__cls` if it's already present in the data, matching C++ Task behavior.
