### Locked Decisions

**Partial failure reporting:**
- validateAccount returns separate `imapResult` and `smtpResult` sub-objects
- `success: true` requires both IMAP and SMTP to pass
- When both fail, IMAP error takes priority at the top-level `error` and `errorType` fields
- On success, IMAP capabilities are included in `imapResult`
- Result shape:
  ```js

    success: boolean,
    error?: string,           // prefixed with "IMAP: " or "SMTP: "
    errorType?: string,       // propagated from failing sub-test
    identifier?: string,      // from MX matching or null
    imapResult: {
      success: boolean,
      error?: string,
      errorType?: string,
      capabilities?: string[]  // present on imapResult.success
    },
    smtpResult: {
      success: boolean,
      error?: string,
      errorType?: string
    },
    imapServer: { hostname: string, port: number },
    smtpServer: { hostname: string, port: number }
  }
  ```

**MX matching location:**
- MX-regex matching lives inside validateAccount only
- MX resolution fails silently — if DNS times out or fails, skip MX matching and continue
- MX resolution runs concurrently with IMAP+SMTP tests via `tokio::join!()`
- Single 15-second timeout wraps the entire `tokio::join!()`

- Connect + Auth + NOOP — use lettre's `test_connection()` (EHLO + auth + NOOP)
- Connect-only mode when no credentials provided
- Return shape: `{ success, error?, errorType? }` — no EHLO extensions

**Error type extension:**
- SMTP uses the same errorType set as IMAP: `connection_refused`, `timeout`, `tls_error`, `auth_failed`, `unknown`

- Error messages include protocol prefix at top level: "IMAP: ..." / "SMTP: ..."
- Sub-result error messages omit the prefix
- No `dns_error` type — MX resolution fails silently

## Summary

Phase 3 implements `testSMTPConnection` and `validateAccount` in `app/mailcore-rs/`, completing the Rust replacement of the C++ N-API addon's full 5-function API surface. The two new modules (`smtp.rs` and `validate.rs`) follow the exact same patterns established by Phase 1 (`provider.rs`) and Phase 2 (`imap.rs`).

For SMTP testing, the `lettre` crate (v0.11.x) is the standard Rust SMTP library. It provides `AsyncSmtpTransport<Tokio1Executor>` with three connection constructors matching the required modes: `relay()` for direct TLS, `starttls_relay()` for STARTTLS upgrade, and `builder_dangerous()` with `Tls::None` for clear connections. Critically, lettre has built-in support for `Mechanism::Xoauth2` — the XOAUTH2 token format (`user=X\x01auth=Bearer Y\x01\x01`) is generated internally. The `test_connection()` method issues an SMTP NOOP after auth, confirming session health beyond what the C++ addon did.

For `validateAccount`, three async operations run concurrently via `tokio::join!()`: the IMAP connection test (calling into Phase 2's `imap.rs` internals), the SMTP connection test, and MX DNS resolution. The hickory-resolver crate (formerly trust-dns-resolver) provides `Resolver::builder_tokio()` + `mx_lookup(domain)` returning an `MxLookup` iterable. MX exchange hostnames are extracted via `record.exchange().to_string()`. MX resolution failure is silent — DNS errors suppress the identifier field, not the validation result.

The critical constraint is the TLS stack: lettre must be configured with `default-features = false` and `features = ["smtp-transport", "tokio1-rustls", "ring"]` to use rustls exclusively, avoiding native-tls which would introduce OpenSSL symbols conflicting with Electron's BoringSSL.

**Primary recommendation:** Add `smtp.rs` (SMTP connection testing) and `validate.rs` (concurrent validation + MX) as new modules. Use lettre 0.11 with `tokio1-rustls` + `ring` features. Use hickory-resolver 0.24 with `tokio` feature for MX lookups. Call Phase 2's IMAP internal functions directly from `validate.rs` — no re-implementation needed.

## Architecture Patterns

### Recommended Module Structure

```
app/mailcore-rs/src/
  lib.rs             # Phase 1: module root + pub mod declarations
  provider.rs        # Phase 1: providerForEmail, registerProviders
  imap.rs            # Phase 2: testIMAPConnection + pub(crate) do_test_imap()
  smtp.rs            # Phase 3: testSMTPConnection — NEW
  validate.rs        # Phase 3: validateAccount — NEW
```

Phase 3 adds `smtp.rs` and `validate.rs`. Both are declared in `lib.rs` as `pub mod smtp; pub mod validate;`.

`validate.rs` calls `imap::do_test_imap()` (the internal function from Phase 2). Phase 2 must expose its core logic as `pub(crate)` — this is the integration between phases.

### Pattern 1: lettre AsyncSmtpTransport — Three Connection Modes

**What:** lettre provides three factory methods matching the three required connection modes. All return `AsyncSmtpTransport<Tokio1Executor>` with a `test_connection()` method.

**When to use:** Route on the `connectionType` option field.

```rust
// Source: lettre docs.rs/lettre/latest/lettre/transport/smtp/struct.AsyncSmtpTransport.html
// + lettre/examples/tokio1_smtp_starttls.rs
use lettre::{AsyncSmtpTransport, AsyncTransport, Tokio1Executor};
use lettre::transport::smtp::authentication::{Credentials, Mechanism};
use lettre::transport::smtp::client::{Tls, TlsParameters};

async fn build_smtp_transport(
    hostname: &str,
    port: u16,
    connection_type: &str,
    username: Option<&str>,
    password: Option<&str>,
    oauth2_token: Option<&str>,
) -> Result<AsyncSmtpTransport<Tokio1Executor>, lettre::transport::smtp::Error> {

    let builder = match connection_type {
        "tls" => AsyncSmtpTransport::<Tokio1Executor>::relay(hostname)?,
        "starttls" => AsyncSmtpTransport::<Tokio1Executor>::starttls_relay(hostname)?,
        "clear" => {
            // builder_dangerous() = no TLS defaults; must set Tls::None explicitly
            AsyncSmtpTransport::<Tokio1Executor>::builder_dangerous(hostname)
                .tls(Tls::None)
        }
        _ => AsyncSmtpTransport::<Tokio1Executor>::relay(hostname)?, // default to TLS
    };

    let builder = builder.port(port);

    // Authentication
    let builder = if let Some(token) = oauth2_token {
        let user = username.unwrap_or("");
        let creds = Credentials::new(user.to_string(), token.to_string());
        builder
            .credentials(creds)
            .authentication(vec![Mechanism::Xoauth2])
    } else if let (Some(user), Some(pass)) = (username, password) {
        let creds = Credentials::new(user.to_string(), pass.to_string());
        builder
            .credentials(creds)
            .authentication(vec![Mechanism::Plain, Mechanism::Login])
    } else {
        builder // no credentials — connect-only mode
    };

    Ok(builder.build())
}
```

**CRITICAL:** `relay()` and `starttls_relay()` use the port from `relay()` defaults (465 and 587 respectively) unless overridden with `.port(port)`. Always call `.port(port)` to honor the caller-specified port.

### Pattern 2: lettre test_connection() — NOOP-Based Health Check

**What:** `AsyncSmtpTransport::test_connection()` performs EHLO + optional AUTH + NOOP to verify the SMTP session is healthy without sending a message. This is the preferred approach for connection testing.

```rust
// Source: docs.rs/lettre/latest/lettre/transport/smtp/struct.AsyncSmtpTransport.html
use lettre::{AsyncSmtpTransport, Tokio1Executor};

async fn test_smtp(transport: &AsyncSmtpTransport<Tokio1Executor>) -> bool {
    transport.test_connection().await.unwrap_or(false)
}
```

**CRITICAL:** `test_connection()` returns `Result<bool, Error>`, NOT `Result<(), Error>`. The bool indicates whether the connection test succeeded. Always unwrap with `unwrap_or(false)` for error→failure mapping, not `is_ok()`.

### Pattern 3: XOAUTH2 with lettre

**What:** XOAUTH2 is a built-in mechanism in lettre. The `Credentials` struct stores username and access token; lettre generates the SASL response internally.

```rust
// Source: lettre/src/transport/smtp/authentication.rs (verified from source)
// XOAUTH2 format generated internally: "user={user}\x01auth=Bearer {token}\x01\x01"
use lettre::transport::smtp::authentication::{Credentials, Mechanism};

fn make_xoauth2_transport(
    hostname: &str,
    port: u16,
    username: &str,
    access_token: &str,
) -> Result<AsyncSmtpTransport<Tokio1Executor>, lettre::transport::smtp::Error> {
    let creds = Credentials::new(
        username.to_string(),    // authentication_identity (the email/user)
        access_token.to_string() // secret (the OAuth2 bearer token)
    );

    AsyncSmtpTransport::<Tokio1Executor>::relay(hostname)
        .map(|b| b
            .port(port)
            .credentials(creds)
            .authentication(vec![Mechanism::Xoauth2])
            .build()
        )
}
```

**Do NOT** build the XOAUTH2 SASL string manually. Unlike Phase 2's IMAP (which required a custom `Authenticator` trait), lettre handles this internally via `Mechanism::Xoauth2`.

### Pattern 4: validateAccount with tokio::join!()

**What:** Three async futures run concurrently. `tokio::join!()` polls all three simultaneously. The outer `tokio::time::timeout` enforces the 15-second limit on the entire concurrent block.

```rust
// Source: tokio docs + locked CONTEXT.md decision
use tokio::time::{timeout, Duration};

#[napi(js_name = "validateAccount")]
pub async fn validate_account(opts: ValidateAccountOptions) -> napi::Result<AccountValidationResult> {
    let result = timeout(
        Duration::from_secs(15),
        async {
            tokio::join!(
                test_imap_internal(&opts),    // calls imap::do_test_imap()
                test_smtp_internal(&opts),    // calls smtp logic
                resolve_mx_records(&opts),    // calls hickory-resolver
            )
        }
    ).await;

    match result {
        Ok((imap_result, smtp_result, mx_identifier)) => {
            build_validation_result(imap_result, smtp_result, mx_identifier, &opts)
        }
        Err(_timeout) => Ok(AccountValidationResult {
            success: false,
            error: Some("Validation timed out after 15 seconds".to_string()),
            error_type: Some("timeout".to_string()),
            identifier: None,
            imap_result: SubResult { success: false, error: Some("timeout".to_string()), ..Default::default() },
            smtp_result: SubResult { success: false, error: Some("timeout".to_string()), ..Default::default() },
            imap_server: ServerInfo { hostname: opts.imap_hostname.clone().unwrap_or_default(), port: opts.imap_port.unwrap_or(993) },
            smtp_server: ServerInfo { hostname: opts.smtp_hostname.clone().unwrap_or_default(), port: opts.smtp_port.unwrap_or(587) },
        })
    }
}
```

**CRITICAL:** `tokio::join!()` is a macro that takes expressions, not closures. All three futures must be pinnable in the current scope. If Phase 2's `do_test_imap()` takes an options struct by reference, all three futures sharing `opts` is fine because `tokio::join!()` polls them on the same thread (no `Send` crossing required).

### Pattern 5: hickory-resolver MX Lookup

**What:** `Resolver::builder_tokio()` creates an async resolver using the system DNS configuration. `mx_lookup(domain)` returns `MxLookup`. Iterating yields `MX` records; `record.exchange().to_string()` gives the hostname string.

```rust
// Source: docs.rs/hickory-resolver/latest/hickory_resolver/ + MxLookup docs
use hickory_resolver::Resolver;

async fn resolve_mx_records(domain: &str) -> Vec<String> {
    let resolver = match Resolver::builder_tokio() {
        Ok(builder) => builder.build(),
        Err(_) => return Vec::new(), // fail silently
    };

    match resolver.mx_lookup(domain).await {
        Ok(mx_lookup) => mx_lookup
            .iter()
            .map(|record| record.exchange().to_string().to_lowercase())
            // strip trailing dot that DNS names have ("google.com." → "google.com")
            .map(|host| host.trim_end_matches('.').to_string())
            .collect(),
        Err(_) => Vec::new(), // fail silently per CONTEXT.md decision
    }
}
```

**Note:** DNS names returned by hickory-resolver include a trailing dot (`.`) per DNS protocol format. Always strip with `.trim_end_matches('.')` before matching against MX patterns from `providers.json`.

**Note:** `Resolver::builder_tokio()` must be called from inside a tokio runtime context. Since this runs inside a `#[napi]` async fn, the tokio runtime is always active.

### Pattern 6: MX-Regex Matching Against providers.json Patterns

**What:** The `provider.rs` module already stores MX patterns from `providers.json` as `mx_match_patterns: Vec<String>`. Phase 3 implements the matching logic in `validate.rs` using these stored patterns.

```rust
// Source: provider.rs existing structure + providers.json format
use regex::{Regex, RegexBuilder};

fn identifier_from_mx_records(mx_records: &[String]) -> Option<String> {
    let providers_lock = crate::provider::PROVIDERS.read().ok()?;
    let providers = providers_lock.as_ref()?;

    for provider in providers {
        for pattern in &provider.mx_match_patterns {
            let anchored = format!("^{}$", pattern);
            if let Ok(regex) = RegexBuilder::new(&anchored)
                .case_insensitive(true)
                .build()
            {
                for mx in mx_records {
                    if regex.is_match(mx) {
                        return Some(provider.identifier.clone());
                    }
                }
            }
        }
    }
    None
}
```

**Note:** This function accesses `provider::PROVIDERS` (the global singleton). The `validate.rs` module needs `use crate::provider` access. Expose `PROVIDERS` and the `mx_match_patterns` field with `pub(crate)` visibility in `provider.rs`.

### Anti-Patterns to Avoid

- **Using lettre default features:** `lettre = "0.11"` without `default-features = false` pulls in `native-tls`, which links OpenSSL on Linux and crashes the Electron process. Always use `default-features = false`.
- **Calling `relay()` without `.port()`:** `relay()` defaults to port 465; calling `.port(port)` is required to honor the caller's port.
- **Forgetting ring crypto provider:** `tokio1-rustls` without `ring` or `aws-lc-rs` panics at first TLS connection attempt.
- **Building XOAUTH2 token manually for lettre:** Unlike Phase 2's async-imap (which required a custom `Authenticator`), lettre's `Mechanism::Xoauth2` builds the SASL token internally. Do not duplicate the encoding logic.
- **Trailing dot in MX hostnames:** `record.exchange().to_string()` returns "google.com." not "google.com". Trim before regex matching.
- **Creating Resolver outside tokio runtime:** `Resolver::builder_tokio()` requires a tokio runtime. Always call inside `async fn`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SMTP STARTTLS upgrade | Manual `EHLO`/`STARTTLS`/TLS handshake TCP state machine | `lettre::AsyncSmtpTransport::starttls_relay()` | STARTTLS for SMTP requires careful state management post-TLS; lettre handles pipelining, extension negotiation, and downgrade protection |
| XOAUTH2 SASL for SMTP | Custom `format!("user={}\x01auth=Bearer {}\x01\x01", ...)` builder | `lettre::Mechanism::Xoauth2` with `Credentials::new(user, token)` | lettre generates the SASL initial-response string internally with correct encoding and base64 if required by server |
| SMTP connection health check | Manual `NOOP\r\n` write + response read | `lettre::AsyncSmtpTransport::test_connection()` | test_connection() handles pre-send connection validation, pool interaction, and error mapping correctly |
| DNS MX record resolution | Raw UDP socket DNS query construction | `hickory_resolver::Resolver::mx_lookup()` | MX record parsing involves multi-section DNS responses, priority sorting, CNAME chasing; hickory handles all of this |
| tokio concurrency for 3 futures | Spawning 3 tasks + channels | `tokio::join!(f1, f2, f3)` | join! polls all three without spawning tasks; simpler, no Send bounds required, no channel overhead |

**Key insight:** lettre abstracts the entire SMTP protocol state machine, which is substantially more complex than IMAP for connection testing because it involves extension negotiation (EHLO response parsing), authentication mechanism selection (server may offer PLAIN but not XOAUTH2), and STARTTLS downgrade protection. Never hand-roll any part of this.

## Common Pitfalls

### Pitfall 1: lettre Default Features Pull in native-tls

**What goes wrong:** `lettre = "0.11"` or `lettre = { version = "0.11" }` activates `native-tls` (it's a default feature). On Linux, native-tls links against OpenSSL. Electron bundles BoringSSL. Symbol conflicts cause the `.node` addon to fail loading.
**Why it happens:** lettre's default feature set is oriented toward standalone Rust programs, not embedded N-API addons inside Electron.
**How to avoid:** Always `default-features = false`. Only request `smtp-transport`, `tokio1-rustls`, `ring`, `pool`, `hostname`.
**Warning signs:** `undefined symbol: SSL_*` or `undefined symbol: EVP_*` in the Electron crash log during addon load.

### Pitfall 2: ring Crypto Provider Missing

**What goes wrong:** `tokio1-rustls` does not automatically select a crypto provider. First TLS connection attempt panics with: `called \`Option::unwrap()\` on a \`None\` value` from inside rustls, or `no process-level CryptoProvider available`.
**Why it happens:** lettre 0.11 split the deprecated `rustls-tls` feature into separate TLS + provider features. The `ring` or `aws-lc-rs` feature must be explicitly enabled.
**How to avoid:** Add `"ring"` to lettre's features list. Ring requires no C build toolchain.
**Warning signs:** Runtime panic on the first `test_connection()` call, not at compilation time.

### Pitfall 3: relay() Port Default Mismatch

**What goes wrong:** `AsyncSmtpTransport::relay("smtp.gmail.com")` defaults to port 465. If the caller passes port 587 (STARTTLS), using `relay()` + `.port(587)` will try direct TLS on port 587, which may fail. More commonly, if the developer forgets `.port()`, Gmail port 993 (IMAP) gets passed to `relay()` but the SMTP connection goes to 465.
**Why it happens:** `relay()` and `starttls_relay()` have their own default ports.
**How to avoid:** Always explicitly call `.port(port)` after the factory method, regardless of connection type.
**Warning signs:** `Connection refused` or `TLS handshake error` when port does not match security mode.

### Pitfall 4: MX Hostname Trailing Dot

**What goes wrong:** `record.exchange().to_string()` returns `"google.com."` (with trailing dot). providers.json MX patterns like `google\\.com` do not end in `\\.`, so the anchored regex `^google\.com$` does NOT match `"google.com."`.
**Why it happens:** DNS Name objects follow RFC 1035 which uses a trailing dot to denote absolute names.
**How to avoid:** `.trim_end_matches('.')` on every MX hostname before regex matching.
**Warning signs:** MX matching returns no identifier even for known providers like gmail; `identifier` is always null.

### Pitfall 5: tokio::join!() vs tokio::try_join!()

**What goes wrong:** `tokio::try_join!()` short-circuits on the first `Err`, cancelling the other futures. If one of `test_imap()`, `test_smtp()`, or `resolve_mx()` returns an Err, the other two stop. This means partial results are impossible with `try_join!()`.
**Why it happens:** The CONTEXT.md design returns separate `imapResult` and `smtpResult` even when one fails — both must always complete.
**How to avoid:** Use `tokio::join!()` (not `try_join!()`). Each branch returns `Result<T, E>` and the outer join returns `(Result<T, E>, Result<T, E>, Result<T, E>)` — all three always complete.
**Warning signs:** SMTP result is missing from validateAccount response when IMAP fails; only IMAP error is present.

### Pitfall 6: napi-rs js_name for validateAccount and testSMTPConnection

**What goes wrong:** `#[napi]` on `async fn validate_account()` auto-converts to camelCase: `validateAccount` (correct). `async fn test_smtp_connection()` auto-converts to `testSmtpConnection` (WRONG — C++ export is `testSMTPConnection`).
**Why it happens:** napi-rs camelCase conversion lowercases consecutive uppercase letters: `SMTP` → `smtp` → `testSmtpConnection`.
**How to avoid:** Use `#[napi(js_name = "testSMTPConnection")]` explicitly. This was the same issue as `testIMAPConnection` in Phase 2.
**Warning signs:** JavaScript `TypeError: napi.testSMTPConnection is not a function` at runtime; the wrapper module routing call fails.

### Pitfall 7: lettre Connection Pool Interaction with test_connection()

**What goes wrong:** When the `pool` feature is enabled (recommended), `test_connection()` behavior changes: the connection is NOT closed afterward (it's recycled). If the pool is dropped before `test_connection()` completes, a deadlock or resource leak may occur.
**Why it happens:** Pool management is automatic but the connection pool holds background tasks.
**How to avoid:** Create the transport, call `test_connection()`, then drop the transport. The transport must remain in scope until the await completes. Do not spawn `test_connection()` into a background task and drop the transport.

### lettre Transport Construction

```rust
// Source: lettre/examples/tokio1_smtp_starttls.rs + lettre docs
use lettre::{AsyncSmtpTransport, AsyncTransport, Tokio1Executor};
use lettre::transport::smtp::authentication::{Credentials, Mechanism};
use lettre::transport::smtp::client::Tls;

async fn do_test_smtp_connection(opts: &SMTPConnectionOptions) -> std::result::Result<SMTPConnectionResult, Box<dyn std::error::Error + Send + Sync>> {
    let conn_type = opts.connection_type.as_deref().unwrap_or("tls");
    let port = opts.port as u16;

    let base_builder = match conn_type {
        "tls" => AsyncSmtpTransport::<Tokio1Executor>::relay(&opts.hostname)?,
        "starttls" => AsyncSmtpTransport::<Tokio1Executor>::starttls_relay(&opts.hostname)?,
        "clear" => AsyncSmtpTransport::<Tokio1Executor>::builder_dangerous(&opts.hostname).tls(Tls::None),
        _ => AsyncSmtpTransport::<Tokio1Executor>::relay(&opts.hostname)?,
    }
    .port(port);

    let transport = match (opts.oauth2_token.as_deref(), opts.username.as_deref(), opts.password.as_deref()) {
        (Some(token), Some(user), _) => {
            let creds = Credentials::new(user.to_string(), token.to_string());
            base_builder
                .credentials(creds)
                .authentication(vec![Mechanism::Xoauth2])
                .build()
        }
        (_, Some(user), Some(pass)) if !user.is_empty() => {
            let creds = Credentials::new(user.to_string(), pass.to_string());
            base_builder
                .credentials(creds)
                .authentication(vec![Mechanism::Plain, Mechanism::Login])
                .build()
        }
        _ => base_builder.build(),
    };

    let ok = transport.test_connection().await?;
    Ok(SMTPConnectionResult {
        success: ok,
        error: if ok { None } else { Some("SMTP server rejected the connection".to_string()) },
        error_type: if ok { None } else { Some("unknown".to_string()) },
    })
}
```

### hickory-resolver MX Lookup

```rust
// Source: docs.rs/hickory-resolver/latest/hickory_resolver/ + MxLookup docs
use hickory_resolver::Resolver;

pub(crate) async fn resolve_mx(domain: &str) -> Vec<String> {
    let resolver = match Resolver::builder_tokio() {
        Ok(b) => b.build(),
        Err(_) => return Vec::new(),
    };

    match resolver.mx_lookup(domain).await {
        Ok(mx_lookup) => mx_lookup
            .iter()
            .map(|r| r.exchange().to_string())
            .map(|h| h.trim_end_matches('.').to_lowercase())
            .collect(),
        Err(_) => Vec::new(),
    }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `trust-dns-resolver` | `hickory-resolver` | Rebranded at v0.24 (2023) | Same API, new crate name. Use hickory-resolver directly |
| `lettre::*rustls-tls` features | `lettre::tokio1-rustls` + explicit `ring` or `aws-lc-rs` | lettre 0.11.x | Must add `ring` feature explicitly; old feature still compiles but logs deprecation warning |
| Manual STARTTLS state machine | `lettre::AsyncSmtpTransport::starttls_relay()` | lettre 0.10+ (stable) | No custom TCP code needed for SMTP STARTTLS |

**Deprecated/outdated:**
- `trust-dns-resolver`: Renamed to `hickory-resolver`. The old crate name still works as a re-export shim but should not be used in new code.
- `lettre::tokio1-rustls-tls`: Deprecated in lettre 0.11; replaced by `tokio1-rustls` + explicit crypto provider feature.
- `lettre` with `default-features = true` in Electron contexts: Always broken due to native-tls.

## Open Questions

1. **lettre TLS certificate validation with rustls-platform-verifier**
   - What we know: Phase 2 uses `rustls-platform-verifier` via `tokio-rustls` directly. lettre uses its own internal TLS stack when features are `tokio1-rustls` + `ring`.
   - What's unclear: lettre's `TlsParameters::new_rustls(domain)` uses the `ring` provider. It may use webpki-roots rather than the OS certificate store. `rustls-platform-verifier` is listed as an optional dependency of lettre but its integration point isn't documented in the builder API.
   - Recommendation: Accept webpki-roots (Mozilla root store) for SMTP TLS for Phase 3. This differs slightly from the IMAP path which uses rustls-platform-verifier (OS root store), but is acceptable for connection testing. If enterprise CA support is required, it's deferred to IMPR-02.
   - Confidence: MEDIUM (verified that lettre lists rustls-platform-verifier as optional dep, but integration via builder API is not clearly documented)

2. **validate.rs access to provider.rs internals (mx_match_patterns)**
   - What we know: `provider.rs` stores `mx_match_patterns` in the `Provider` struct, which is private. The `PROVIDERS` LazyLock is also private.
   - What's unclear: The exact visibility change needed (make `PROVIDERS` and `Provider.mx_match_patterns` `pub(crate)`).
   - Recommendation: Add `pub(crate)` accessor function `get_mx_identifier_for_records(mx_records: &[String]) -> Option<String>` to `provider.rs`. This encapsulates the matching logic and avoids exposing internal Provider struct fields.
