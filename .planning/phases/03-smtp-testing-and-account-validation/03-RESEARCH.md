## Summary

Phase 3 adds two functions to the napi-rs addon: `testSMTPConnection` and `validateAccount`. Both functions are new files (`smtp.rs` and `validator.rs`) that follow the exact patterns established in Phase 2's `imap.rs`. Phase 3 is primarily an assembly phase — the hard infrastructure problems (napi-rs async, rustls TLS, tokio runtime) are already solved in Phases 1 and 2. The key new work is the lettre SMTP library API, DNS MX resolution via hickory-resolver, and composing the IMAP and SMTP tests concurrently with `tokio::join!()`.

The lettre crate (0.11.19, current as of 2025) is the only maintained async SMTP library in the Rust ecosystem with XOAUTH2 support. Its `AsyncSmtpTransport` API maps cleanly to the three connection types: TLS (Wrapper), STARTTLS (Required), and clear (None). XOAUTH2 authentication uses `Credentials::new(username, oauth_token)` with `Mechanism::Xoauth2` — the token is passed as the "password" field and lettre formats the SASL XOAUTH2 challenge response internally. The `test_connection()` method issues a NOOP command and returns `Result<bool, Error>`, exactly matching the success criterion.

For DNS MX resolution, hickory-resolver 0.25.2 (renamed from trust-dns-resolver in v0.24) provides an async Tokio-native resolver with `mx_lookup()`. The `validateAccount` function uses MX records only for provider identifier matching in the result — it does not use MX records to choose which server to connect to (the C++ implementation receives explicit server parameters and resolves MX separately for the identifier field). The IMAP and SMTP connection tests run via `tokio::join!(imap_fut, smtp_fut)` inside a single napi async fn, satisfying VALD-01.

**Primary recommendation:** Add `lettre` with `tokio1-rustls-tls` feature (no native-tls) and `hickory-resolver` to Cargo.toml. Implement `smtp.rs` mirroring Phase 2's `imap.rs` structure, then `validator.rs` composing both with `tokio::join!()`.

### Pattern 1: lettre TLS/STARTTLS/Clear Connection Type Selection

**What:** Use `builder_dangerous()` for full control over port, TLS, and credentials. Map the `connectionType` string to lettre's `Tls` enum variants.

**When to use:** All three connection types in `testSMTPConnection`.

**Example:**
```rust
// Source: verified from lettre docs.rs 0.11.19 + lettre/src/transport/smtp/async_transport.rs
use lettre::{
    transport::smtp::{
        authentication::{Credentials, Mechanism},
        client::{Tls, TlsParameters},
        AsyncSmtpTransport,
    },
    AsyncTransport, Tokio1Executor,
};
use std::time::Duration;

async fn build_smtp_transport(
    hostname: &str,
    port: u16,
    connection_type: &str,
    credentials: Option<Credentials>,
    auth_mechanism: Option<Mechanism>,
) -> Result<AsyncSmtpTransport<Tokio1Executor>, lettre::transport::smtp::Error> {
    let tls = match connection_type {
        "tls" => {
            // SMTPS: immediate TLS wrapper (port 465)
            let tls_params = TlsParameters::new_rustls(hostname.to_owned())?;
            Tls::Wrapper(tls_params)
        }
        "starttls" => {
            // STARTTLS: plain connect then mandatory upgrade (port 587)
            let tls_params = TlsParameters::new_rustls(hostname.to_owned())?;
            Tls::Required(tls_params)
        }
        _ => {
            // clear: no TLS at all (port 25)
            Tls::None
        }
    };

    let mut builder = AsyncSmtpTransport::<Tokio1Executor>::builder_dangerous(hostname)
        .port(port)
        .tls(tls)
        .timeout(Some(Duration::from_secs(15)));  // SMTP-05: 15s timeout

    if let Some(creds) = credentials {
        builder = builder.credentials(creds);
    }
    if let Some(mechanism) = auth_mechanism {
        builder = builder.authentication(vec![mechanism]);
    }

    Ok(builder.build())
}
```

### Pattern 2: XOAUTH2 Credentials

**What:** For XOAUTH2, pass the OAuth2 bearer token as the "password" field of `Credentials::new()` and specify `Mechanism::Xoauth2`. Lettre internally formats the SASL XOAUTH2 response as `user={username}\x01auth=Bearer {token}\x01\x01`.

**When to use:** When `oauth2Token` is non-empty in the input options.

**Example:**
```rust
// Source: verified from lettre/src/transport/smtp/authentication.rs source code analysis
use lettre::transport::smtp::authentication::{Credentials, Mechanism};

fn build_credentials(
    username: &str,
    password: &str,
    oauth2_token: &str,
) -> Option<(Credentials, Mechanism)> {
    if !oauth2_token.is_empty() {
        // XOAUTH2: username = email address, password = the OAuth2 bearer token
        // lettre formats: "user={username}\x01auth=Bearer {token}\x01\x01"
        Some((
            Credentials::new(username.to_owned(), oauth2_token.to_owned()),
            Mechanism::Xoauth2,
        ))
    } else if !password.is_empty() {
        // Standard password auth: PLAIN or LOGIN depending on server preference
        // lettre's default mechanism order is [Plain, Login] which handles both
        Some((
            Credentials::new(username.to_owned(), password.to_owned()),
            Mechanism::Plain,
        ))
    } else {
        None
    }
}
```

**Critical:** The `Mechanism::Xoauth2` enum variant must be explicitly passed via `.authentication(vec![Mechanism::Xoauth2])`. Without this, lettre tries PLAIN/LOGIN first, which will fail on servers that only accept XOAUTH2.

### Pattern 3: test_connection() NOOP Test

**What:** After building the transport, call `transport.test_connection().await` which issues an SMTP NOOP command. Returns `Ok(true)` if connected successfully. This exactly matches the C++ `session->connectIfNeeded() + loginIfNeeded()` pattern.

**When to use:** Core of `testSMTPConnection` — the NOOP proves the connection is live.

**Example:**
```rust
// Source: verified from lettre docs.rs AsyncSmtpTransport::test_connection docs
pub async fn test_smtp_connection_inner(
    hostname: &str,
    port: u16,
    connection_type: &str,
    username: &str,
    password: &str,
    oauth2_token: &str,
) -> Result<bool, String> {
    let creds_and_mechanism = build_credentials(username, password, oauth2_token);

    let transport = build_smtp_transport(
        hostname, port, connection_type,
        creds_and_mechanism.as_ref().map(|(c, _)| c.clone()),
        creds_and_mechanism.as_ref().map(|(_, m)| *m),
    ).await.map_err(|e| format!("SMTP transport error: {}", e))?;

    // Issues NOOP command — proves connection + auth are both working
    // This mirrors C++ connectIfNeeded() + loginIfNeeded()
    transport.test_connection().await.map_err(|e| format!("SMTP connection failed: {}", e))
}
```

**Note:** `test_connection()` does NOT issue NOOP if using connection pool (pool feature keeps connection open). Since we add `pool` to default features in lettre, be aware of this behavior. For a pure test function, not sending a message, the NOOP is still issued and the pool connection is recycled after.

### Pattern 4: tokio::join!() for Concurrent IMAP + SMTP

**What:** Inside a `#[napi] pub async fn validate_account(...)`, use `tokio::join!()` to run the IMAP and SMTP test futures concurrently. Both futures execute on the tokio runtime that napi-rs manages. This satisfies VALD-01.

**When to use:** `validateAccount` function.

**Example:**
```rust
// Source: napi.rs/docs/concepts/async-fn + tokio docs
use napi_derive::napi;
use napi::Result;

#[napi(js_name = "validateAccount")]
pub async fn validate_account(opts: ValidateAccountOptions) -> Result<AccountValidationResult> {
    // Concurrently test IMAP and SMTP — satisfies VALD-01
    let (imap_result, smtp_result) = tokio::join!(
        crate::imap::test_imap_connection_inner(
            &opts.imap_hostname.unwrap_or_default(),
            opts.imap_port.unwrap_or(993) as u16,
            "tls",  // default connection type for validator
            &opts.email,
            &opts.password.clone().unwrap_or_default(),
            &opts.oauth2_token.clone().unwrap_or_default(),
        ),
        test_smtp_connection_inner(
            &opts.smtp_hostname.unwrap_or_default(),
            opts.smtp_port.unwrap_or(587) as u16,
            "starttls",  // default connection type for validator
            &opts.email,
            &opts.password.clone().unwrap_or_default(),
            &opts.oauth2_token.clone().unwrap_or_default(),
        ),
    );

    // VALD-02: resolve MX records for provider identifier
    let domain = opts.email.split('@').last().unwrap_or("").to_owned();
    let identifier = resolve_provider_identifier(&domain).await;

    build_validation_result(imap_result, smtp_result, identifier, &opts)
}
```

**Critical:** `tokio::join!()` runs both futures concurrently on the SAME tokio task (not spawning new OS threads). Both SMTP and IMAP connections proceed in parallel without blocking each other. This is the correct pattern — do NOT use `tokio::task::spawn_blocking` or OS threads for this.

### Pattern 5: DNS MX Resolution for Provider Identifier

**What:** Use hickory-resolver to look up MX records for the email domain, then compare MX hostnames against providers.json `mx-match` patterns to determine the provider identifier. This replicates the C++ `MCAccountValidator::identifier()` behavior.

**When to use:** Inside `validateAccount`, after the IMAP/SMTP tests.

**Example:**
```rust
// Source: verified from hickory-resolver docs.rs 0.25.2
use hickory_resolver::{Resolver, config::{ResolverConfig, ResolverOpts}};

async fn resolve_mx_records(domain: &str) -> Vec<String> {
    // Use system resolver config — matches what the C++ implementation does
    let resolver = match Resolver::builder_tokio() {
        Ok(builder) => builder.build(),
        Err(_) => return vec![],
    };

    match resolver.mx_lookup(domain).await {
        Ok(mx_records) => mx_records
            .iter()
            .map(|mx| mx.exchange().to_lowercase().to_string())
            // Strip trailing dot that DNS adds: "aspmx.l.google.com." -> "aspmx.l.google.com"
            .map(|s| s.trim_end_matches('.').to_owned())
            .collect(),
        Err(_) => vec![],  // DNS failure: return empty, don't fail validation
    }
}

async fn resolve_provider_identifier(domain: &str) -> Option<String> {
    let mx_records = resolve_mx_records(domain).await;

    // Access the provider database (already loaded in Phase 1)
    // This reuses the RwLock<Option<Vec<Provider>>> singleton from provider.rs
    let guard = crate::provider::PROVIDERS.read().unwrap();
    let providers = guard.as_ref()?;

    for provider in providers {
        for mx_pattern in &provider.mx_match_patterns {
            let anchored = format!("^{}$", mx_pattern);
            if let Ok(re) = regex::RegexBuilder::new(&anchored)
                .case_insensitive(true)
                .build()
            {
                if mx_records.iter().any(|mx| re.is_match(mx)) {
                    return Some(provider.identifier.clone());
                }
            }
        }
    }
    None
}
```

**DNS failure behavior:** The C++ implementation resolves MX in the background; if DNS fails, `identifier()` returns null. Mirror this: DNS failure returns `None` for identifier, does not fail the overall `validateAccount` result.

### Pattern 6: C++ Result Shape (VALD-03 contract)

**What:** The `AccountValidationResult` TypeScript interface (from `types/index.d.ts`) has:
```typescript
interface AccountValidationResult {
  success: boolean;
  error?: string;
  identifier?: string;
  imapServer?: { hostname: string; port: number };
  smtpServer?: { hostname: string; port: number };
}
```

The Rust struct must match this exactly. All optional fields that are absent in the C++ output must be `Option<T>` in Rust (napi-rs maps `Option<T>` to `field?: T` in TypeScript).

**When to use:** `AccountValidationResult` napi object struct.

**Example:**
```rust
// Source: Direct analysis of app/mailcore/src/napi/napi_validator.cpp OnOK() + types/index.d.ts
use napi_derive::napi;

#[napi(object)]
pub struct ServerResult {
    pub hostname: String,
    pub port: u32,
}

#[napi(object)]
pub struct AccountValidationResult {
    pub success: bool,
    pub error: Option<String>,
    pub identifier: Option<String>,
    pub imap_server: Option<ServerResult>,
    pub smtp_server: Option<ServerResult>,
}
```

**Critical napi-rs naming:** napi-rs auto-converts `imap_server` (snake_case) to `imapServer` (camelCase) in TypeScript. The generated `.d.ts` will have `imapServer?: { hostname: string; port: number }` matching the existing interface. Do NOT use `#[napi(js_name)]` on object fields — the auto-conversion is correct here.

### Anti-Patterns to Avoid

- **Using lettre with default features:** Default features include `native-tls` which introduces OpenSSL — conflicts with Electron's BoringSSL. Always `default-features = false` with explicit `tokio1-rustls-tls`.
- **Using Tls::Opportunistic for STARTTLS:** `Opportunistic` silently falls back to plaintext if STARTTLS fails. Use `Tls::Required` for SMTP-02 — it fails with an error if TLS negotiation fails, matching the C++ behavior.
- **Not calling `.authentication(vec![Mechanism::Xoauth2])`:** Without this, lettre tries PLAIN first. Some servers will reject PLAIN when XOAUTH2 is intended, causing auth failures that are hard to diagnose.
- **Spawning tokio tasks for IMAP/SMTP in validateAccount:** Use `tokio::join!()` not `tokio::spawn()`. Spawned tasks need `'static` lifetime bounds and complicate ownership. `join!()` runs both futures on the same task.
- **Using trust-dns-resolver crate name:** This is the old name. As of v0.24, the crate is `hickory-resolver`. The old name still redirects on crates.io but is considered deprecated — use the new name.
- **Trailing dot on DNS MX results:** `mx_lookup()` returns fully-qualified hostnames with trailing dots (e.g., `aspmx.l.google.com.`). Strip the trailing dot before comparing against providers.json MX patterns.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SMTP connection with TLS | Custom TCP + TLS handshake | lettre `AsyncSmtpTransport` | SMTP state machine (EHLO, AUTH, NOOP) is complex; lettre handles all states correctly |
| STARTTLS upgrade | Manual `STARTTLS` command then TLS wrap | lettre `Tls::Required` | Stream upgrade timing and error handling have subtle edge cases |
| XOAUTH2 SASL challenge | Custom base64 encoding of XOAUTH2 format | lettre `Mechanism::Xoauth2` | XOAUTH2 has specific encoding: `user={u}\x01auth=Bearer {t}\x01\x01`; easy to get wrong |
| DNS MX lookup | Raw UDP/TCP DNS socket code | `hickory-resolver` | DNS wire format, retries, CNAME following, TTL — do not hand-roll |
| SMTP NOOP connection test | Send a real message to /dev/null | `transport.test_connection()` | NOOP is the correct way; sending a test message creates server-side state |
| Concurrent async futures | Manual thread spawn + channel | `tokio::join!()` | join! is the idiomatic Tokio pattern for concurrent independent futures |

**Key insight:** The SMTP protocol has a complex state machine (connect → EHLO → AUTH → NOOP → QUIT). Any custom implementation will miss edge cases like pipelining, multi-line responses, and authentication challenge sequences. lettre encodes 10+ years of SMTP compatibility fixes.

## Common Pitfalls

### Pitfall 1: lettre Default Features Include native-tls

**What goes wrong:** Adding `lettre = "0.11"` to Cargo.toml (without `default-features = false`) pulls in `native-tls` → OpenSSL symbols → conflicts with Electron's BoringSSL → addon crashes on load.

**Why it happens:** lettre's default feature set is designed for general use cases where native-tls is fine. The project has a hard constraint against OpenSSL from Phase 1.

**How to avoid:** Always use:
```toml
lettre = { version = "0.11", default-features = false, features = ["smtp-transport", "tokio1", "tokio1-rustls-tls", "hostname"] }
```

**Warning signs:** `cargo tree | grep openssl` outputs anything after adding lettre.

### Pitfall 2: Tls::Opportunistic vs Tls::Required for STARTTLS

**What goes wrong:** Using `Tls::Opportunistic` for the STARTTLS case (SMTP-02). If the server doesn't advertise STARTTLS, the connection proceeds in plaintext without error. The test appears to "pass" but credentials are sent unencrypted.

**Why it happens:** The name "Opportunistic" sounds like "try STARTTLS if available" — which is correct semantically, but wrong for a connection *test* that should fail if TLS is unavailable.

**How to avoid:** Use `Tls::Required` for the `"starttls"` connection type in testSMTPConnection. Match the C++ behavior: `ConnectionTypeStartTLS` fails if TLS cannot be negotiated.

**Warning signs:** Tests against a clear-text-only server "pass" when using STARTTLS settings.

### Pitfall 3: XOAUTH2 Mechanism Must Be Explicitly Specified

**What goes wrong:** Setting `Credentials::new(username, oauth_token)` without calling `.authentication(vec![Mechanism::Xoauth2])`. lettre's default mechanism order is `[Plain, Login]`. The server receives a PLAIN auth attempt with the OAuth2 token as the "password" — this fails with an auth error that looks like a wrong password.

**Why it happens:** lettre treats XOAUTH2 as an opt-in mechanism. The Credentials struct alone doesn't specify which mechanism to use.

**How to avoid:** Always pair XOAUTH2 credentials with `.authentication(vec![Mechanism::Xoauth2])`.

**Warning signs:** `SMTP authentication failed: 535 5.7.8 Username and Password not accepted` when the token is correct.

### Pitfall 4: DNS Trailing Dot on MX Hostnames

**What goes wrong:** `mx_lookup("gmail.com").await` returns `MxRecord` with exchange `"aspmx.l.google.com."` (trailing dot). Comparing this against providers.json pattern `aspmx\\.l\\.google\\.com` fails because `\.` doesn't match `.` at the end of a string with a trailing dot.

**Why it happens:** DNS fully-qualified domain names (FQDNs) include a trailing dot by convention. The `MxLookup` iterator returns the raw DNS response which includes the trailing dot.

**How to avoid:**
```rust
// Strip trailing dot before matching
let mx_host = mx.exchange().to_string();
let mx_host = mx_host.trim_end_matches('.');
```

**Warning signs:** Provider identifier is `None` for Gmail accounts even when `aspmx.l.google.com` should match the `gmail` provider.

### Pitfall 5: Hickory-Resolver Tokio Runtime Conflict

**What goes wrong:** Calling `Resolver::builder_tokio()` inside a `#[napi] pub async fn` (which is already inside the napi-rs tokio runtime) can panic with "Cannot start a runtime from within a runtime" in some configurations.

**Why it happens:** Some versions of hickory-resolver create an internal tokio runtime when constructing the resolver. Creating a nested tokio runtime inside an existing one panics.

**How to avoid:** Use `Resolver::builder_tokio()` which uses the ambient tokio runtime (no new runtime created). This is specifically safe inside tokio async contexts. The `Resolver::new(...)` constructor (which creates an internal runtime) should NOT be used.

**Warning signs:** `thread panicked with 'Cannot start a runtime from within a runtime'` on the first `validateAccount` call.

### Pitfall 6: timeout() is Per-Command, Not Per-Connection

**What goes wrong:** Setting `.timeout(Some(Duration::from_secs(15)))` on the lettre transport builder and expecting that the total connection + auth + NOOP sequence completes within 15 seconds. In reality, the `timeout` field controls each individual SMTP command's timeout, not the total operation.

**Why it happens:** The SMTP protocol is multi-step: TCP connect, TLS handshake, EHLO, AUTH, NOOP, QUIT. Each step uses the configured timeout independently. A slow server could time out on each step individually, accumulating to much more than 15 seconds total.

**How to avoid:** Wrap the entire `test_smtp_connection_inner()` call in `tokio::time::timeout()`:
```rust
use tokio::time::{timeout, Duration};

let result = timeout(
    Duration::from_secs(15),
    test_smtp_connection_inner(hostname, port, connection_type, username, password, oauth_token),
).await;

match result {
    Ok(Ok(connected)) => connected,
    Ok(Err(e)) => return Err(format!("SMTP error: {}", e)),
    Err(_elapsed) => return Err("SMTP connection timed out after 15 seconds".to_owned()),
}
```

This is the same pattern used in Phase 2's `testIMAPConnection` with async-imap. Apply it consistently.

**Warning signs:** Hangs longer than 15 seconds against a nonexistent SMTP host.

### Pitfall 7: connection_type Default in C++ is "tls" but validateAccount Uses Different Defaults

**What goes wrong:** The C++ `testSMTPConnection` defaults to `connectionType = "tls"` when the field is absent. But the C++ `validateAccount` (`napi_validator.cpp`) hardcodes `ConnectionTypeTLS` for both IMAP and SMTP server params. However, looking at onboarding-helpers.ts, the caller passes explicit connection settings — the "default" in validateAccount is largely academic.

**Why it happens:** The C++ AccountValidator has its own connection logic independent of testSMTPConnection. The Rust implementation must match the caller's expectation: `validateAccount` should use whatever connection type is implied by the port (993 → TLS, 587 → STARTTLS) or explicitly provided.

**How to avoid:** Inspect the opts: if `connectionType` is not provided in `validateAccount` options, infer from port (993/465 = tls, 587/143 = starttls). Match the C++ result: the Rust implementation calls `test_imap_connection_inner` and `test_smtp_connection_inner` from `validator.rs`, passing the connection type through.

**Warning signs:** validateAccount succeeds with explicit params but fails when connection type is not specified.

### Pitfall 8: validateAccount IMAP Default Port (993) vs C++ Hardcoded ConnectionTypeTLS

**What goes wrong:** Looking at `napi_validator.cpp`, when `imapHostname` is provided, the C++ creates a `NetService` with `ConnectionTypeTLS` hardcoded. The port from the opts (`imapPort` defaulting to 993) is used. This means validateAccount always tests IMAP with TLS regardless of the caller's connection type preference.

**Why it happens:** The C++ AccountValidator API was designed to accept server params that include an explicit connection type. The napi binding simplifies this by always using TLS.

**How to avoid:** In `validator.rs`, call `test_imap_connection_inner` with `"tls"` as the connection type and use the `imapPort` from opts (defaulting to 993). Same for SMTP: use `"starttls"` and port 587 as the default. This matches the C++ behavior.

// src/smtp.rs
// Source: mirrors Phase 2 imap.rs structure + lettre docs.rs 0.11.19

use std::time::Duration;
use napi::Result;
use napi_derive::napi;
use lettre::{
    transport::smtp::{
        authentication::{Credentials, Mechanism},
        client::{Tls, TlsParameters},
        AsyncSmtpTransport,
    },
    AsyncTransport, Tokio1Executor,
};
use tokio::time::timeout;

#[napi(object)]
pub struct SmtpConnectionResult {
    pub success: bool,
    pub error: Option<String>,
}

pub async fn test_smtp_inner(
    hostname: &str,
    port: u16,
    connection_type: &str,
    username: &str,
    password: &str,
    oauth2_token: &str,
) -> std::result::Result<bool, String> {
    let build_tls = || -> std::result::Result<Tls, String> {
        match connection_type {
            "tls" => {
                let p = TlsParameters::new_rustls(hostname.to_owned())
                    .map_err(|e| format!("TLS params error: {}", e))?;
                Ok(Tls::Wrapper(p))
            }
            "starttls" => {
                let p = TlsParameters::new_rustls(hostname.to_owned())
                    .map_err(|e| format!("TLS params error: {}", e))?;
                Ok(Tls::Required(p))
            }
            _ => Ok(Tls::None),
        }
    };

    let tls = build_tls()?;

    let mut builder = AsyncSmtpTransport::<Tokio1Executor>::builder_dangerous(hostname)
        .port(port)
        .tls(tls)
        .timeout(Some(Duration::from_secs(15)));

    if !oauth2_token.is_empty() {
        builder = builder
            .credentials(Credentials::new(username.to_owned(), oauth2_token.to_owned()))
            .authentication(vec![Mechanism::Xoauth2]);
    } else if !password.is_empty() {
        builder = builder
            .credentials(Credentials::new(username.to_owned(), password.to_owned()));
        // default mechanisms (Plain, Login) are fine for password auth
    }

    let transport = builder.build();

    // Wrap in outer timeout for total operation (see pitfall 6)
    let test = timeout(
        Duration::from_secs(15),
        transport.test_connection(),
    ).await;

    match test {
        Ok(Ok(connected)) => Ok(connected),
        Ok(Err(e)) => Err(format!("SMTP error: {}", e)),
        Err(_) => Err("SMTP connection timed out after 15 seconds".to_owned()),
    }
}

#[napi(js_name = "testSMTPConnection")]
pub async fn test_smtp_connection(opts: TestSmtpOptions) -> Result<SmtpConnectionResult> {
    let result = test_smtp_inner(
        &opts.hostname,
        opts.port as u16,
        opts.connection_type.as_deref().unwrap_or("tls"),
        &opts.username.unwrap_or_default(),
        &opts.password.unwrap_or_default(),
        &opts.oauth2_token.unwrap_or_default(),
    ).await;

    match result {
        Ok(true) => Ok(SmtpConnectionResult { success: true, error: None }),
        Ok(false) => Ok(SmtpConnectionResult { success: false, error: Some("Connection test returned false".into()) }),
        Err(e) => Ok(SmtpConnectionResult { success: false, error: Some(e) }),
    }
}

}

use napi::Result;
use napi_derive::napi;
use hickory_resolver::Resolver;
use regex::RegexBuilder;

pub struct ServerResult {
    pub hostname: String,
    pub port: u32,
}

#[napi(object)]
pub struct AccountValidationResult {
    pub success: bool,
    pub error: Option<String>,
    pub identifier: Option<String>,
    pub imap_server: Option<ServerResult>,  // -> imapServer in TypeScript
    pub smtp_server: Option<ServerResult>,  // -> smtpServer in TypeScript
}

#[napi(object)]
pub struct ValidateAccountOptions {
    pub email: String,
    pub password: Option<String>,
    pub oauth2_token: Option<String>,
    pub imap_hostname: Option<String>,
    pub imap_port: Option<u32>,
    pub smtp_hostname: Option<String>,
    pub smtp_port: Option<u32>,
}

async fn resolve_provider_identifier(domain: &str) -> Option<String> {
    let resolver = Resolver::builder_tokio().ok()?.build();

    let mx_records: Vec<String> = match resolver.mx_lookup(domain).await {
        Ok(records) => records
            .iter()
            .map(|mx| mx.exchange().to_string())
            .map(|s| s.trim_end_matches('.').to_lowercase())  // strip trailing DNS dot
            .collect(),
        Err(_) => return None,
    };

    // Reuse provider database from Phase 1 provider.rs
    let guard = crate::provider::PROVIDERS.read().ok()?;
    let providers = guard.as_ref()?;

    for provider in providers {
        for pattern in &provider.mx_match_patterns {
            let anchored = format!("^{}$", pattern);
            if let Ok(re) = RegexBuilder::new(&anchored).case_insensitive(true).build() {
                if mx_records.iter().any(|mx| re.is_match(mx)) {
                    return Some(provider.identifier.clone());
                }
            }
        }
    }
    None
}

#[napi(js_name = "validateAccount")]
pub async fn validate_account(opts: ValidateAccountOptions) -> Result<AccountValidationResult> {
    let imap_host = opts.imap_hostname.clone().unwrap_or_default();
    let imap_port = opts.imap_port.unwrap_or(993) as u16;
    let smtp_host = opts.smtp_hostname.clone().unwrap_or_default();
    let smtp_port = opts.smtp_port.unwrap_or(587) as u16;
    let password = opts.password.clone().unwrap_or_default();
    let oauth_token = opts.oauth2_token.clone().unwrap_or_default();

    // VALD-01: concurrent IMAP + SMTP tests
    let (imap_result, smtp_result) = tokio::join!(
        crate::imap::test_imap_inner(
            &imap_host, imap_port, "tls",
            &opts.email, &password, &oauth_token,
        ),
        crate::smtp::test_smtp_inner(
            &smtp_host, smtp_port, "starttls",
            &opts.email, &password, &oauth_token,
        ),
    );

    // VALD-02: resolve MX for provider identifier (non-blocking: failure = None)
    let domain = opts.email.split('@').last().unwrap_or("").to_owned();
    let identifier = resolve_provider_identifier(&domain).await;

    // VALD-03: result shape matching C++ output
    let imap_success = imap_result.as_ref().map(|r| *r).unwrap_or(false);
    let smtp_success = smtp_result.as_ref().map(|r| *r).unwrap_or(false);
    let success = imap_success && smtp_success;

    let error = if !success {
        let imap_err = imap_result.err().unwrap_or_default();
        let smtp_err = smtp_result.err().unwrap_or_default();
        Some(format!("IMAP: {} | SMTP: {}", imap_err, smtp_err).trim_matches('|').trim().to_owned())
    } else {
        None
    };

    Ok(AccountValidationResult {
        success,
        error,
        identifier,
        imap_server: Some(ServerResult {
            hostname: imap_host.clone(),
            port: imap_port as u32,
        }),
        smtp_server: Some(ServerResult {
            hostname: smtp_host.clone(),
            port: smtp_port as u32,
        }),
    })
}

| Old Approach | Current Approach | When Changed | Impact |

- `async-smtp` crate: archived, unmaintained, no rustls support. Do not use.
- `lettre 0.9` and earlier: blocking only, no async. Do not use.
- `trust-dns-resolver` crate name: use `hickory-resolver` v0.25 instead.
- `Resolver::new()` from hickory-resolver: creates internal runtime, panics inside tokio context. Use `Resolver::builder_tokio()`.

## Open Questions

1. **Does lettre's `test_connection()` authenticate before issuing NOOP, or just check TCP connectivity?**
   - What we know: From source analysis, `test_connection()` issues SMTP NOOP. The transport builder is configured with credentials. During normal connection establishment (which NOOP triggers), lettre negotiates TLS and AUTH before sending commands.
   - What's unclear: Whether NOOP specifically triggers full AUTH when using connection pool vs. non-pool configuration.
   - Recommendation: Disable the `pool` feature in Cargo.toml for this addon (or leave it enabled — pool feature simply caches the connection). The NOOP test will authenticate as part of EHLO/AUTH exchange before NOOP. Verify with a live server test against Gmail/Outlook.

2. **Should `validateAccount` pass connection type through from opts or hardcode TLS/STARTTLS defaults?**
   - What we know: The C++ napi_validator.cpp hardcodes `ConnectionTypeTLS` for both IMAP and SMTP. The `validateAccount` TypeScript interface in `types/index.d.ts` does NOT include `connectionType` parameters — only hostname and port.
   - What's unclear: Whether the port alone is sufficient to determine TLS mode (993 → TLS, 587 → STARTTLS, etc.) or if an explicit connection type field should be added.
   - Recommendation: Infer connection type from port in `validator.rs`: port 993 or 465 → `"tls"`, port 587 or 143 → `"starttls"`, other → `"tls"` as safe default. This matches the C++ behavior without adding a new parameter to the TypeScript interface.

3. **Does hickory-resolver 0.25 work in the napi-rs tokio context without "runtime within runtime" panic?**
   - What we know: `Resolver::builder_tokio()` is documented to use the ambient tokio runtime. The napi-rs tokio runtime is a standard multi-thread tokio runtime.
   - What's unclear: Whether hickory-resolver's internal design actually avoids creating a new runtime in 0.25.2. GitHub issue #2142 documents this panic for earlier versions.
   - Recommendation: Use `Resolver::builder_tokio()` (not `Resolver::new()`). If panic occurs in testing, fall back to spawning resolver construction in a separate `tokio::task::spawn_blocking` call. Flag for early validation in Phase 3 implementation.
