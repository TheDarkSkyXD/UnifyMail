## Summary

Phase 4.2 is a targeted fix-and-verify phase closing two audit gaps (INTG-VALIDATE-USERNAME, FLOW-VALIDATE-RUNTIME). The work has two independent parts that must happen in sequence: (1) split `ValidateAccountOptions` in Rust from shared credentials to per-protocol credentials, then fix the TS call site to pass all required fields; (2) extend the existing Electron integration test to prove the E2E field mapping is correct.

The current state is well-understood from direct code inspection. The Rust struct at `validate.rs:85-99` has `username: Option<String>` and `password: Option<String>` shared fields. The `build_imap_opts` and `build_smtp_opts` helpers at lines 172-193 both read these shared fields and pass them unchanged to the protocol-specific options structs. The TS call site at `mailsync-process.ts:441-449` only passes `password` (as `settings.imap_password`) and does not pass `imap_username`, `smtp_username`, `smtp_password`, or security→connectionType mapping.

The fix is surgical: rename struct fields in Rust, update the two builder helpers, update the TS call site with the 3-value security mapping and per-protocol credential lookup, update the existing Rust integration test helper (`validate_opts()` in `smtp_tests.rs`) to use the new field names, and add Electron integration test checks that verify field echo.

**Primary recommendation:** Make all Rust changes (struct rename + builder update + existing test update) in one task, then do the TS call site fix + Electron test extension in a second task — this keeps compilation-breaking changes isolated and testable.

### Pattern 3: TypeScript Security String → connectionType Mapping

**What:** Inline 3-value map in the `test()` method translating Account model security strings to Rust vocabulary.

**Account model values:** `'SSL / TLS'`, `'STARTTLS'`, `'none'`
**Rust vocabulary:** `'tls'`, `'starttls'`, `'clear'`
**Default when undefined:** `undefined` → Rust treats absent `connectionType` as `'tls'`

```typescript
// Source: pattern derived from CONTEXT.md locked decision
// Inline in test() method — not a helper function
function mapSecurity(security: string | undefined): string | undefined {
  if (security === 'SSL / TLS') return 'tls';
  if (security === 'STARTTLS') return 'starttls';
  if (security === 'none') return 'clear';
  return undefined; // undefined → Rust defaults to TLS
}
```

The call in `test()` applies this inline:

### Anti-Patterns to Avoid

- **Manual edits to `index.d.ts`:** It is auto-generated by napi-rs. Any manual edit will be overwritten on the next `npm start` or `npm run build:rust`. Never edit it directly — let the Rust struct change drive the type update.
- **Rust-side username fallback:** The locked decision places the fallback (`|| account.emailAddress`) on the TS side only. Do not add fallback logic to the Rust struct or builder helpers.
- **Separate helper function for security mapping:** The locked decision says inline in `test()`. A helper would be cleaner but contradicts the explicit decision.
- **Using `check()` for async calls without await:** The existing synchronous `check()` in the Electron test swallows async errors silently. Async checks must use `await` inside an async context or chain `.then()/.catch()` explicitly.

## Common Pitfalls

### Pitfall 1: Forgetting to Update `validate_opts()` Test Helper

**What goes wrong:** After renaming `username`/`password` to `imap_username`/`imap_password`/`smtp_username`/`smtp_password` in the Rust struct, the existing `validate_opts()` helper in `smtp_tests.rs` (lines 557-570) still uses the old field names. The code will not compile.

**Why it happens:** The struct change is in `validate.rs`; the test helper is in `smtp_tests.rs`. Rust compiler will flag the old field names as errors, but it's easy to miss this as a separate file to update.

**How to avoid:** Update `validate_opts()` in `smtp_tests.rs` in the same task as the struct rename. The helper currently sets `username` and `password`; it must set all four per-protocol fields.

**Warning signs:** `cargo test` fails with "struct `ValidateAccountOptions` has no field named `username`".

### Pitfall 2: napi-rs Field Naming: snake_case in Rust, camelCase in JS

**What goes wrong:** A developer writes `imap_username` in Rust and then in the TS call site writes `imap_username` instead of `imapUsername`. The field silently passes `undefined` (napi-rs ignores unknown camelCase keys).

**Why it happens:** The camelCase conversion is implicit and easy to forget when thinking in Rust naming.

**How to avoid:** After the Rust build, check the auto-generated `index.d.ts` to confirm the field names before writing TS code. The pattern is: any `_` in Rust becomes camelCase in JS.

**Warning signs:** Rust tests pass (they use the Rust struct directly), but the Electron integration test fails because credentials come through as `null`/`undefined`.

### Pitfall 3: Electron Integration Test async/await Scope

**What goes wrong:** Adding a `validateAccount` call inside the synchronous `check()` helper. Since `check()` does not `await`, the Promise rejection is unhandled and the check appears to pass (the `fn()` call returns without throwing synchronously).

**Why it happens:** The existing `check()` helper pattern is synchronous. `validateAccount` returns a Promise. A silent "pass" when the async work never ran is the most confusing failure mode.

**How to avoid:** Either convert the `app.whenReady().then()` callback to `async` and `await` the result, or use a separate `checkAsync()` pattern. The chosen approach should be clearly documented in the test file.

**Warning signs:** Check shows PASS in the log but the total count stays at 7 (async check never incremented results).

### Pitfall 4: `oauth2_token` Field Name in JS

**What goes wrong:** Passing `oauth2Token` in JS (camelCase) but the existing `index.d.ts` already has `oauth2Token?: string | null`. This field is unchanged, so no risk of mismatch — but it must not be accidentally renamed.

**Why it happens:** The split rename of `username`/`password` might cause a developer to also rename `oauth2_token` unnecessarily.

**How to avoid:** Leave `oauth2_token` in Rust unchanged. The `index.d.ts` will continue to show `oauth2Token` (the camelCase form that already works).

### Pitfall 5: `imapConnectionType` vs `connectionType` in `index.d.ts`

**What goes wrong:** Looking at the old `ValidateAccountOptions` in `index.d.ts` (line 143-153), the field is already `imapConnectionType` (not `connectionType`). The struct already uses `imap_connection_type` in Rust. When adding `imap_username`, the JS name will be `imapUsername` — consistent with this existing pattern.

**Why it happens:** Confusion when comparing the validate struct field names with the IMAP/SMTP standalone structs (which use `connection_type` → `connectionType` without protocol prefix). The validate struct already has protocol-prefixed names for hostname/port/connectionType.

**How to avoid:** The convention for `ValidateAccountOptions` is already protocol-prefixed. The new credential fields follow this same pattern: `imap_username` → `imapUsername`, `smtp_password` → `smtpPassword`.
