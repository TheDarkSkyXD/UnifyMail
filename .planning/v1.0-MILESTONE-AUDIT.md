### Integration Gaps

#### Gap 1: validateAccount call site omits `username` (Medium)

**ID:** INTG-VALIDATE-USERNAME
**Affected:** INTG-02, VALD-01
**File:** `app/src/browser/mailsync-process.ts` ~line 441

The `validateAccount()` call passes `password` and `oauth2Token` but not `username`. In Rust, `opts.username` defaults to `None` → empty string for IMAP LOGIN and empty `user=` in XOAUTH2 SASL token. OAuth2 accounts fail IMAP auth via N-API fast path. The catch fallback to `_spawnAndWait('test')` rescues the flow.

**Fix needed:** Add `username: settings.imap_username` to the call site.

#### Gap 2: validateAccount call site omits connection type (Medium)

**ID:** INTG-VALIDATE-CONNTYPE
**Affected:** INTG-02, VALD-01, VALD-03
**File:** `app/src/browser/mailsync-process.ts` ~line 441

`imapConnectionType` and `smtpConnectionType` not passed. Both default to `None` → `"tls"`. Accounts using STARTTLS (port 143/587) fail the N-API fast path.

**Fix needed:** Map `settings.imap_security`/`settings.smtp_security` to connection type strings.

#### Gap 3: macOS CI runner label (Medium, CI)

**ID:** INTG-CI-RUNNER
**Affected:** SCAF-03
**File:** `.github/workflows/build-macos.yaml`

Uses `macos-15-intel` runner which may not be a valid GitHub-hosted runner label (documented Intel runners: `macos-13`, `macos-14-large`). The darwin-x64 target may fail to find a runner. Requires CI test run or label correction.

## Audit Conclusion

All 27 v1 requirements are verified as SATISFIED at the phase level. The Rust napi-rs addon correctly implements all 5 functions with comprehensive test coverage. CI workflows build for all 5 targets with shared smoke tests covering all exports. All C++ code has been deleted.

**Blockers preventing milestone completion:**

1. **Phase 4.2 unstarted** — assigned milestone phase with no plans, no work, no verification
2. **validateAccount N-API fast path broken** — the integration checker confirmed the TypeScript call site in `mailsync-process.ts` is missing `username` and connection type fields, causing the Rust validation path to fail for OAuth2 and STARTTLS accounts (the two gaps Phase 4.2 was created to close)

The app functions correctly because the catch fallback to `_spawnAndWait('test')` rescues failed N-API calls. The gaps are in the TypeScript → Rust wiring, not in the Rust implementation itself.
