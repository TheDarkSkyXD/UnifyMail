# Phase 4.2: validateAccount Integration Verification - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix and verify the field mapping between `mailsync-process.ts`'s `test()` method and the Rust `validateAccount` function. The call site currently omits username, smtp_password, and connection type fields. The Rust API needs per-protocol credential fields to match the Account model. An integration test must prove the E2E credential flow is correct.

Gap Closure: Closes INTG-VALIDATE-USERNAME, FLOW-VALIDATE-RUNTIME from v1.0 audit.

</domain>

<decisions>
## Implementation Decisions

### Credential field mapping
- Split Rust `ValidateAccountOptions` into per-protocol credentials: `imap_username`/`smtp_username` and `imap_password`/`smtp_password` (remove the single `username` and `password` fields)
- Keep single `oauth2_token` field — OAuth2 tokens are account-wide, never per-protocol
- The TS call site in `mailsync-process.ts` passes `settings.imap_username`, `settings.smtp_username`, `settings.imap_password`, `settings.smtp_password` individually
- When `imap_username` or `smtp_username` is undefined/empty, the TS call site falls back to `account.emailAddress` (not the Rust side)

### Connection type passing
- The TS call site maps `settings.imap_security`/`settings.smtp_security` to Rust's `connectionType` vocabulary before calling validateAccount
- Mapping: `'SSL / TLS'` → `'tls'`, `'STARTTLS'` → `'starttls'`, `'none'` → `'clear'`
- Mapping is inline in the `test()` method (not a separate helper function) — it's a 3-value map used in one place
- When security setting is undefined, TLS is the default (Rust default behavior, no change needed)

### Integration test approach
- Electron integration test added to the existing `test/electron-integration-test.js` file (extends the existing 7-check pattern)
- Test creates an Account-like object with known field values, runs the same mapping logic `mailsync-process.ts` uses, then calls `validateAccount` via the N-API binding
- Verification uses error response field echo: call with fake host (`'test-imap.invalid'`), assert `imapServer.hostname`, `imapServer.port`, `smtpServer.hostname`, `smtpServer.port` match inputs — proves field mapping without needing a live mail server
- Test covers both the TS-side mapping logic (security → connectionType, username fallback) AND the N-API boundary crossing

### Claude's Discretion
- Exact Rust struct field naming (e.g., `imap_username` vs `imapUsername` in Rust — napi-rs handles camelCase conversion)
- Whether `build_imap_opts`/`build_smtp_opts` helpers need refactoring for the split fields
- Test assertion count and structure within electron-integration-test.js
- Error message format assertions in the test

</decisions>
